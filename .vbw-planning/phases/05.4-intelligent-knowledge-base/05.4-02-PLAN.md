## Objective
Extend the DOCX parser to tag each paragraph with its document zone (header/footer/body/table_cell/cover), implement heuristic blueprint detection from completed mapping plans, and collect style-to-mappability hints. This runs in the Python sanitization service and provides structural intelligence that the Node.js backend stores in the KB (Plan 05.4-01 models). Acceptance criteria #6, #8 are partially addressed here (detection logic; storage wiring in Plan 03).

## Context
- @sanitization-service/app/services/docx_parser.py -- DocxParserService.parse() extracts paragraphs, tables, images, sections. _extract_paragraphs() loops over doc.paragraphs (body only). _extract_sections() gets header/footer paragraphs. Currently no zone tagging -- paragraphs from body, headers, footers, and table cells all land in flat lists.
- @sanitization-service/app/models/docx.py -- DocxParagraph has text, style_name, heading_level, alignment, runs. DocxSection has header_paragraphs and footer_paragraphs. DocxTable has rows -> cells -> paragraphs. Zone info is available structurally but not tagged on individual paragraphs.
- @sanitization-service/app/models/adapter.py -- MappingPlan and MappingEntry models used for blueprint detection input. FIELD_MARKER_MAP defines known field-to-marker mappings.
- @sanitization-service/app/services/analysis_prompt.py -- _build_doc_structure_section() emits paragraphs with index + heading level. Zone tags will enhance this output in Plan 05.4-03.
- Decision #9: Heuristic blueprint detection -- structural patterns are deterministic, no LLM needed
- Decision #5: Style-to-zone hints with filtering (60%+ prompt noise reduction target)

### Task 3: Blueprint Detection Heuristics
**File(s):** `sanitization-service/app/services/blueprint_detector.py` (new file)
**Action:** Create
**Details:**
- Create new module with `detect_blueprints(mapping_plan: MappingPlan, doc_structure: DocxStructure) -> list[dict]`
- Implement three heuristic rules (Decision #9):
  1. **Loop detection**: Find consecutive mapping entries where marker_type is "table_row_loop" or "control_flow" and they share a loop variable prefix (e.g., "finding.*"). Group them into a loop blueprint with ordered markers list.
  2. **Group detection**: Find entries sharing a gwField prefix (e.g., "finding.title", "finding.severity_rt") within the same zone and within 10 paragraph indices of each other. Group them as a "group" blueprint.
  3. **Conditional detection**: Find "control_flow" entries (if/endif patterns) that bracket other mapping entries. Create conditional blueprint.
- Return format: list of dicts matching BlueprintPattern fields: `{ templateType, zone, patternType, markers: [{gwField, markerType}], anchorStyle }`
- Extract anchorStyle from the first paragraph's style_name in the pattern
- Log detected blueprints at INFO level for debugging
**Commit:** `feat(kb): implement heuristic blueprint detection from mapping plans`

### Task 4: Style Hint Collection
**File(s):** `sanitization-service/app/services/blueprint_detector.py`
**Action:** Modify (append to file created in Task 3)
**Details:**
- Add function `collect_style_hints(doc_structure: DocxStructure, mapping_plan: MappingPlan) -> list[dict]`:
  1. Build a set of mapped paragraph indices from mapping_plan.entries
  2. For each paragraph in doc_structure.paragraphs:
     - Get its style_name and zone
     - If paragraph index is in mapped set: increment mappedCount for that (style, zone)
     - Else: increment skippedCount
  3. Return list of dicts matching StyleHint fields: `{ templateType: mapping_plan.template_type, styleName, zone, mappedCount, skippedCount }`
- This function is deterministic and fast (no LLM, pure counting)
- Handles edge cases: paragraphs with no style_name use "Normal" as default
**Commit:** `feat(kb): add style hint collection from document structure and mapping plan`

### Task 5: Blueprint Detection Endpoint
**File(s):** `sanitization-service/app/routes/adapter.py`, `sanitization-service/app/models/adapter.py`
**Action:** Modify
**Details:**
- Add Pydantic request/response models in adapter.py:
  ```python
  class DetectBlueprintsRequest(BaseModel):
      template_base64: str
      mapping_plan: MappingPlan
      template_type: TemplateType
      language: TemplateLanguage

  class BlueprintResult(BaseModel):
      template_type: str
      zone: str
      pattern_type: str
      markers: list[dict]
      anchor_style: str | None = None

  class StyleHintResult(BaseModel):
      template_type: str
      style_name: str
      zone: str
      mapped_count: int
      skipped_count: int

  class DetectBlueprintsResponse(BaseModel):
      blueprints: list[BlueprintResult]
      style_hints: list[StyleHintResult]
  ```
- Add `POST /adapter/detect-blueprints` endpoint:
  1. Decode template_base64, parse DOCX with DocxParserService (gets zone-tagged structure)
  2. Call `detect_blueprints(mapping_plan, doc_structure)`
  3. Call `collect_style_hints(doc_structure, mapping_plan)`
  4. Return DetectBlueprintsResponse
- Follow existing endpoint patterns (error handling, logging)
**Commit:** `feat(adapter): add POST /detect-blueprints endpoint for structural intelligence`
