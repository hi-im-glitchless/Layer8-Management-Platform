# Phase 5.4 Context: Intelligent Knowledge Base

## User Vision

Evolve the flat mapping KB (TemplateMapping table with text-to-field tuples) into a structural intelligence layer that understands document zones, repetition patterns, structural blueprints, style heuristics, and confidence calibration. Target: auto-map accuracy on familiar template types jumps from ~15/90 to 80%+ without manual corrections.

## Essential Features

### Zone-Aware Storage
- Add `zone` field to TemplateMapping: header|footer|body|table_cell|text_box|cover|unknown
- Existing entries migrate with zone='unknown'
- LLM prompt groups examples by zone so it knows WHERE to place markers

### Repetition Tracking Per Zone
- Track how many times a gwField appears in each zone for a given template type
- Example: "client.short_name appears 8x in body, 2x in headers, 42x in tables for web/en"
- Solves the core problem: LLM mapped only ~8 of 52 client.short_name instances

### Structural Blueprints (New Model: BlueprintPattern)
- Store composite multi-marker structures as reusable patterns
- Blueprint types: loop (for/endfor + inner fields), conditional, group (co-occurring markers)
- Example: findings table blueprint = {%tr for finding in findings %} + V{{loop.index}} + {{finding.title}} + {{r finding.severity_rt}} + {%tr endfor %}
- Biggest accuracy gap is structural markers — they always appear as coordinated groups

### Confidence Calibration (Decay/Boost)
- On correction (Phase 5.3 flow): decay confidence by 0.7x
- On confirmation (unchanged through download): boost by min(1.0, +0.1)
- Wrong mappings sink below threshold and stop appearing in prompts
- Fixes: currently a mapping confirmed once then corrected 5x still shows high usageCount

### Style-Based Heuristics (New Model: StyleHint)
- Track which paragraph styles typically contain placeholders vs boilerplate
- If 'Heading 2' paragraphs are NEVER mapped in web/en templates, skip them in prompt
- Reduces prompt noise by ~60% (122 body paragraphs → ~30 mappable)

### Structured Prompt Context Block
- Replace flat "Previous Successful Mappings" numbered list with structured sections:
  - Zone Distribution (field counts per zone)
  - Blueprints (composite structural patterns)
  - Confidence Notes (high/low confidence entries)
- Far more actionable than flat list for LLM reasoning

## Technical Preferences

### Data Model: Three Prisma Models
1. **TemplateMapping** (evolved): add zone, zoneRepetitionCount, correctionCount fields
2. **BlueprintPattern** (new): templateType, zone, patternType (loop|conditional|group), markers[] (ordered gwField+markerType list), anchorStyle
3. **StyleHint** (new): templateType, styleName, zone, mappedCount, skippedCount

### Population Triggers (Three)
1. **On download** (existing): bulk upsert confirmed mappings — now includes zone data
2. **On correction** (new): immediately decay old entry confidence, create/boost corrected entry
3. **On blueprint detection** (new): after successful analysis, auto-detect blueprints from mapping plan

### Blueprint Detection: Heuristic Rules
- Adjacent table_row_loop markers in same table → loop blueprint
- Consecutive gwFields sharing prefix (finding.*) in same zone → group blueprint
- control_flow markers bracketing other markers → loop+content blueprint
- Pure code logic, no LLM needed — structural patterns are deterministic

### Token Budget for KB Context
- Dynamic: 15-25% of analysis prompt, capped at 1500 tokens
- Priority order: blueprints first, then zone-tagged mappings, then flat mappings
- Prevents KB context from crowding out actual document structure

### Cross-Template-Type Learning
- Primary lookup scoped to (templateType, language) as today
- NEW: If no entries exist for a template type (first use), fall back to entries from other types with 0.7x confidence penalty
- Most field mappings are shared across types; type-specific patterns remain isolated

### Cold Start Strategy
- No KB entries → fall back to reference template pattern matching (existing) + style-based zone inference (new)
- First adaptation is ~40-50% accuracy; each correction and download populates KB for next time
- No seed dataset or manual pre-mapping required

### Direct Mapping Table (Replaces LLM Correction Chat)
- **Remove the correction chat box entirely** from Step 2 (Verify Placeholders)
- **Show editable mapping table** pre-filled with all auto-mapped entries from LLM
- Each row: `#N | [document text] | [GW field / Jinja2 expression]`
- **PDF highlight → new table row**: user highlights text on PDF, row auto-added to table
- **GW field autocomplete**: dropdown/typeahead of GW fields that auto-generates correct Jinja2 syntax based on field type and zone. Raw Jinja2 input also supported for early KB bootstrapping or advanced users
- **Delete row to remove mapping**: X button removes mapping; on regeneration, that location reverts to original text
- **Bidirectional table-PDF sync**: click table row → PDF scrolls to location and highlights it. The table replaces the structure browser as the navigation tool
- **LLM stays for auto-map**: LLM still does Pass 1 + Pass 2 auto-mapping on upload. KB makes auto-map 80%+. Table is for the remaining 20% of corrections
- **Rationale**: Phase 5.3 test proved LLM correction chat loses 70% of corrections (90 typed → 27 applied). Direct table editing is 100% deterministic

## Boundaries

- No LLM-assisted blueprint extraction (heuristic rules only)
- No manual blueprint definition UI
- No real-time KB writes during wizard flow (only on download/correction/detection triggers)
- No global KB across template types (always scoped, with optional cross-type fallback)
- No schema redesign — evolve existing TemplateMapping, add new models alongside
- No LLM correction chat — all corrections through direct mapping table editing

## Acceptance Criteria

1. TemplateMapping has zone, zoneRepetitionCount, correctionCount fields; migration preserves existing data
2. BlueprintPattern model stores and queries composite structural patterns
3. StyleHint model tracks style-to-mappability frequencies
4. Confidence decay (0.7x) fires on correction; boost (+0.1, cap 1.0) fires on confirmation
5. Analysis prompt receives structured KB context block (zones + blueprints + confidence)
6. Blueprint detection runs automatically after successful analysis
7. Cross-type fallback works with 0.7x confidence penalty for first-use template types
8. Style hints filter boilerplate paragraphs from prompt (60%+ reduction)
9. Auto-map accuracy on web/en template reaches 80%+ (measured against known 90-mapping template)
10. Mapping table pre-fills with all auto-mapped entries, supports add/edit/delete
11. GW field autocomplete generates correct Jinja2 syntax; raw input also supported
12. Bidirectional table-PDF sync: click row → scroll PDF, highlight PDF → add row
13. Correction chat removed; all corrections happen through table editing

## Decisions Made

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Zone column on TemplateMapping | Parser already knows zones; minimal schema change |
| 2 | Repetition counts per zone | Fixes the 52-vs-8 client.short_name gap |
| 3 | Composite BlueprintPattern model | Structural markers always appear as groups |
| 4 | Decay/boost confidence calibration | Current usageCount-only approach poisons prompts |
| 5 | Style-to-zone hints with filtering | 60%+ prompt noise reduction |
| 6 | Structured prompt context block | More actionable than flat numbered list |
| 7 | Three-model Prisma evolution | Clear separation of concerns |
| 8 | Three population triggers | Corrections are highest-signal feedback |
| 9 | Heuristic blueprint detection | Structural patterns are deterministic, no LLM needed |
| 10 | Dynamic 15-25% token budget (1500 cap) | Balance KB context vs document structure |
| 11 | Cross-type fallback with 0.7x penalty | Bootstraps new types without polluting established patterns |
| 12 | Reference template cold start | No seed dataset needed |
| 13 | Replace correction chat with direct mapping table | LLM chat lost 70% of corrections in 5.3 testing |
| 14 | GW field autocomplete with raw Jinja2 fallback | Autocomplete for known fields, raw input for KB bootstrapping |
| 15 | Pre-fill table with all auto-mapped entries | Table IS the mapping plan — user edits in place |
| 16 | Delete row to remove mapping (not blank field) | Unambiguous: deleted = restore original text |
| 17 | Bidirectional table-PDF sync | Table replaces structure browser as navigation tool |
| 18 | LLM for auto-map, table for corrections | AI does heavy lifting, human fixes directly |
