## Context
Set up Gotenberg as a Docker container for DOCX-to-PDF conversion, and implement a BullMQ-based background queue in the Node.js backend. LibreOffice (inside Gotenberg) is not thread-safe, so all conversions are queued and processed one at a time. This plan has no dependencies on other Phase 4 plans because it only needs a DOCX file as input (from any source).

## Tasks

### Task 1: Docker Compose with Gotenberg
**Commit:** `feat(docker): add docker-compose.yml with Gotenberg service`
**Files:**
- docker-compose.yml -- new file at project root

**Steps:**
1. Create `docker-compose.yml` at project root with:
   ```yaml
   services:
     gotenberg:
       image: gotenberg/gotenberg:8
       restart: unless-stopped
       ports:
         - "3000:3000"
       command:
         - "gotenberg"
         - "--api-timeout=120s"
         - "--libreoffice-restart-after=10"
         - "--libreoffice-max-queue-size=1"
       healthcheck:
         test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
         interval: 10s
         timeout: 5s
         retries: 3
   ```
2. Key Gotenberg settings:
   - `--libreoffice-max-queue-size=1`: Only one LibreOffice conversion at a time (thread safety)
   - `--libreoffice-restart-after=10`: Restart LibreOffice after 10 conversions (memory leak prevention)
   - `--api-timeout=120s`: Allow up to 2 minutes for large documents
3. Port 3000 is Gotenberg's default. Backend connects via `http://localhost:3000`
4. Add `GOTENBERG_URL` to backend `.env` and `.env.example`:
   ```
   GOTENBERG_URL=http://localhost:3000
   ```
5. Add to `backend/src/config.ts` Zod schema:
   ```typescript
   GOTENBERG_URL: z.string().default('http://localhost:3000'),
   ```

### Task 2: BullMQ PDF Conversion Queue
**Commit:** `feat(pdf): implement BullMQ background queue for PDF conversion`
**Files:**
- backend/src/services/pdfQueue.ts -- new file
- backend/package.json -- add bullmq dependency

**Steps:**
1. Install BullMQ: `npm install bullmq`
2. Create `backend/src/services/pdfQueue.ts`:
   - Create a BullMQ `Queue` named `pdf-conversion` connected to the existing Redis instance
   - Create a BullMQ `Worker` that processes jobs:
     - Read DOCX file path from job data
     - POST the DOCX to Gotenberg's `/forms/libreoffice/convert` endpoint as multipart form data
     - Save the returned PDF to `backend/uploads/documents/{uuid}.pdf`
     - Update job progress and store PDF path in job return value
   - Worker concurrency = 1 (LibreOffice thread safety)
   - Configure job options: `removeOnComplete: { age: 3600 }` (clean up after 1 hour), `removeOnFail: { age: 86400 }` (keep failures for 24 hours)
   - Export `addPdfConversionJob(docxPath: string, originalName: string): Promise<string>` -- adds job, returns job ID
   - Export `getPdfJobStatus(jobId: string): Promise<{ status, progress, pdfPath?, error? }>`
   - Gotenberg multipart request: POST to `${GOTENBERG_URL}/forms/libreoffice/convert` with the DOCX file as form field named `files`
3. Ensure `backend/uploads/documents/` directory is created on startup (add to index.ts alongside existing avatar directory creation)

### Task 3: PDF Conversion API Routes
**Commit:** `feat(pdf): add PDF conversion API endpoints with job tracking`
**Files:**
- backend/src/routes/documents.ts -- new file
- backend/src/index.ts -- mount new router

**Steps:**
1. Create `backend/src/routes/documents.ts` with Router:
   - `POST /api/documents/convert-pdf`:
     - Requires auth (`requireAuth` middleware)
     - Accepts multipart upload via multer (single file, field name `file`)
     - Validates: file exists, extension is `.docx`, MIME type matches, size <= 50MB
     - Saves uploaded DOCX to `backend/uploads/documents/{uuid}.docx`
     - Calls `addPdfConversionJob()` to queue conversion
     - Returns `{ jobId, status: 'queued' }` with 202 Accepted
   - `GET /api/documents/convert-pdf/:jobId`:
     - Requires auth
     - Calls `getPdfJobStatus(jobId)`
     - Returns job status: `{ status: 'queued'|'active'|'completed'|'failed', progress, pdfUrl?, error? }`
     - When completed, `pdfUrl` points to the file served via static route
   - `GET /api/documents/download/:filename`:
     - Requires auth
     - Serves file from `backend/uploads/documents/` with proper content-type
     - Validates filename format (UUID pattern) to prevent path traversal
   - Zod validation on all inputs
2. Configure multer with:
   - Storage: `backend/uploads/documents/`
   - Filename: `${uuid}${ext}` (randomised)
   - File filter: accept only `.docx` and `.pdf` extensions
   - Size limit: 50MB
3. Mount in `backend/src/index.ts`: `app.use('/api/documents', documentsRouter)`
4. Add audit logging for document uploads and conversions
