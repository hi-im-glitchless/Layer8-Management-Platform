## Context
Build the DOCX parsing and generation endpoints in the sanitization-service (FastAPI). This extends the existing Python service with document processing capabilities using python-docx (already in requirements.txt). The parser extracts structured content from uploaded DOCX files; the generator produces DOCX files with Jinja2 placeholders. These endpoints are consumed by the Node.js backend for the template adapter workflow.

**Commit:** `feat(docx): add Pydantic models for DOCX parsing and generation`
**Files:**
- sanitization-service/app/models/docx.py -- new file with all DOCX-related Pydantic models

**Steps:**
1. Create `sanitization-service/app/models/docx.py` with Pydantic models:
   - `DocxParagraph`: text, style_name, heading_level (optional int), alignment, runs (list of `DocxRun`)
   - `DocxRun`: text, bold, italic, underline, font_name, font_size, color
   - `DocxTable`: rows (list of `DocxRow`), style_name
   - `DocxRow`: cells (list of `DocxCell`)
   - `DocxCell`: text, paragraphs (list of `DocxParagraph`), merge_info (optional)
   - `DocxImage`: content_type, width, height, filename, paragraph_index
   - `DocxSection`: header_paragraphs, footer_paragraphs (lists of `DocxParagraph`), page_width, page_height, orientation
   - `DocxStructure` (main response): paragraphs, tables, images, sections, styles (list of style names), metadata (dict)
   - `GenerateDocxRequest`: template_content (str, the Jinja2 template as base64-encoded DOCX), placeholders (dict of placeholder values)
   - `GenerateDocxResponse`: filename (str), size_bytes (int)
2. Export all models from `app/models/__init__.py`

### Task 2: DOCX Parser Service
**Commit:** `feat(docx): implement DOCX parsing service with full structure extraction`
**Files:**
- sanitization-service/app/services/docx_parser.py -- new file

**Steps:**
1. Create `sanitization-service/app/services/docx_parser.py` with class `DocxParserService`:
   - `parse(file_bytes: bytes) -> DocxStructure`: Main entry point
   - Uses `Document(BytesIO(file_bytes))` to load DOCX from bytes
   - `_extract_paragraphs(doc)`: Iterate `doc.paragraphs`, extract text, style name, heading level (from style like "Heading 1" -> level=1), alignment, and runs with formatting (bold, italic, underline, font properties)
   - `_extract_tables(doc)`: Iterate `doc.tables`, for each table extract rows/cells with text and nested paragraphs. Handle merged cells via `cell._tc` XML inspection
   - `_extract_images(doc)`: Iterate `doc.inline_shapes` to find images, extract content type, dimensions, and associated paragraph index. Use `shape._inline.graphic.graphicData` for image data access
   - `_extract_sections(doc)`: Iterate `doc.sections`, extract header/footer paragraphs (both default and first-page), page dimensions, orientation
   - `_extract_styles(doc)`: Collect unique style names used across paragraphs and tables
   - `_extract_metadata(doc)`: Read `doc.core_properties` for author, title, created, modified, etc.
2. Handle edge cases: empty paragraphs, nested tables, missing styles, corrupt/minimal DOCX files
3. Return `DocxStructure` Pydantic model populated with all extracted data

### Task 3: DOCX Generator Service
**Commit:** `feat(docx): implement DOCX generation service with Jinja2 placeholder insertion`
**Files:**
- sanitization-service/app/services/docx_generator.py -- new file

**Steps:**
1. Create `sanitization-service/app/services/docx_generator.py` with class `DocxGeneratorService`:
   - `generate(template_bytes: bytes, context: dict) -> bytes`: Takes a DOCX template (with Jinja2 placeholders) and a data context, returns rendered DOCX bytes
   - Uses `docxtpl` approach: Load DOCX, find Jinja2 `{{ }}` and `{% %}` markers in paragraphs/tables/headers/footers, replace with rendered values while preserving formatting
   - For simple text placeholders: Replace `{{ variable }}` with the context value, preserving the run's formatting (bold, font, size)
   - For rich text (`{{p ...}}`, `{{r ...}}`): These are custom markers from Ghostwriter templates; parse HTML content and insert as formatted DOCX paragraphs/runs
   - For table row loops (`{%tr for ...%}`): Duplicate the template row for each item, replacing placeholders within each copy
   - Write output to BytesIO and return bytes
2. Add `docxtpl>=0.18` and `jinja2>=3.1` to sanitization-service/requirements.txt (docxtpl is the python-docx + Jinja2 integration library that handles `{%tr %}` and `{{r }}` natively)
3. Note: docxtpl handles most of the Ghostwriter template syntax natively (`{%tr for %}`, `{{r }}` for run-level, `{{p }}` for paragraph-level rich text). Leverage its built-in capabilities rather than reimplementing.

### Task 4: DOCX API Routes
**Commit:** `feat(docx): add parse and generate DOCX API endpoints`
**Files:**
- sanitization-service/app/routes/docx.py -- new file
- sanitization-service/app/routes/__init__.py -- add new router
- sanitization-service/app/main.py -- mount new router

**Steps:**
1. Create `sanitization-service/app/routes/docx.py` with APIRouter:
   - `POST /parse-docx`: Accepts `UploadFile` (multipart), validates file is .docx (check extension and MIME type `application/vnd.openxmlformats-officedocument.wordprocessingml.document`), max 50MB. Calls `DocxParserService.parse()`, returns `DocxStructure` JSON
   - `POST /generate-docx`: Accepts JSON body with `GenerateDocxRequest` (base64 template + context dict). Calls `DocxGeneratorService.generate()`, returns the rendered DOCX as `StreamingResponse` with `application/vnd.openxmlformats-officedocument.wordprocessingml.document` content type and `Content-Disposition: attachment`
   - Both endpoints validate input and return proper HTTP error codes (400 for invalid files, 413 for oversized, 422 for malformed)
2. Update `app/routes/__init__.py` to export `docx_router`
3. Update `app/main.py` to mount: `app.include_router(docx_router, tags=["document"])`
