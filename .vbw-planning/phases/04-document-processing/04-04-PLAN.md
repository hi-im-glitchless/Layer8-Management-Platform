## Context
Connect the DOCX generation service (Plan 04-01) with the Ghostwriter data (Plan 04-03) through a Jinja2 rendering pipeline, and wire into the PDF conversion queue (Plan 04-02) for preview generation. The sanitization-service receives a DOCX template and a data context, renders Jinja2 placeholders using docxtpl, and returns the filled document. The Node.js backend orchestrates: fetching GW data, mapping it to template context, calling the Python renderer, and queuing PDF conversion. This is the critical integration that proves the end-to-end document generation pipeline works.

## Tasks

### Task 1: Jinja2 Template Renderer Service
**Commit:** `feat(jinja2): implement template renderer with rich text and custom filters`
**Files:**
- sanitization-service/app/services/template_renderer.py -- new file

**Steps:**
1. Create `sanitization-service/app/services/template_renderer.py` with class `TemplateRendererService`:
   - `render(template_bytes: bytes, context: dict) -> bytes`: Main entry point
   - Uses `docxtpl.DocxTemplate(BytesIO(template_bytes))` to load template
   - Processes the context to convert HTML strings to docxtpl `RichText` objects:
     - For fields ending in `_rt` suffix or known rich text fields (description, impact, recommendation, replication_steps, affected_entities): Parse HTML and convert to `docxtpl.RichText`
     - Use `docxtpl.RichText()` API: `rt.add(text, bold=..., italic=..., url_id=...)` for formatted runs
     - HTML parsing: Use a simple HTML-to-RichText converter that handles `<p>`, `<b>`, `<i>`, `<a>`, `<br>` tags. Strip XHTML namespace attributes (`xmlns="..."`)
   - Register custom Jinja2 filters on the docxtpl template environment:
     - `filter_type(findings, types)`: Filter findings list by finding_type matching any of the given type strings. Used by Internal templates: `findings|filter_type(["Web"])`
   - Render: `tpl.render(context)` then save to BytesIO and return bytes
   - Handle template rendering errors gracefully: undefined variables, missing filters, malformed templates
2. Helper function `html_to_richtext(html_string: str, tpl: DocxTemplate) -> RichText`:
   - Parse HTML using `html.parser.HTMLParser` (stdlib, no extra dependency)
   - Convert `<p>` to paragraph breaks, `<b>` to bold runs, `<i>` to italic, `<a href>` to hyperlinks
   - Strip `<div>` containers and XHTML namespace declarations
   - Return a `docxtpl.RichText` object
3. Helper function `prepare_context(raw_context: dict, tpl: DocxTemplate) -> dict`:
   - Walk the context dictionary recursively
   - Convert HTML strings in known rich text fields to RichText objects
   - For findings list: add `_rt` suffixed versions of each HTML field (e.g., `finding['description_rt'] = html_to_richtext(finding['description'], tpl)`)
   - Add computed fields: `finding['severity_rt']`, `finding['cvss_vector_link_rt']` (CVSS as hyperlink to FIRST.org calculator)

### Task 2: Render Template API Route
**Commit:** `feat(jinja2): add render-template API endpoint`
**Files:**
- sanitization-service/app/routes/docx.py -- add render endpoint
- sanitization-service/app/models/docx.py -- add render request model

**Steps:**
1. Add to `sanitization-service/app/models/docx.py`:
   ```python
   class RenderTemplateRequest(BaseModel):
       template_base64: str  # Base64-encoded DOCX template
       context: dict  # Jinja2 rendering context (template variables)
   ```
2. Add to `sanitization-service/app/routes/docx.py`:
   - `POST /render-template`: Accepts `RenderTemplateRequest` JSON body
     - Decode base64 template to bytes
     - Validate decoded bytes are a valid DOCX (check magic bytes)
     - Call `TemplateRendererService.render(template_bytes, context)`
     - Return rendered DOCX as `StreamingResponse` with proper content type
     - Error handling: 400 for invalid base64/DOCX, 422 for template rendering errors (undefined variables, etc.)
