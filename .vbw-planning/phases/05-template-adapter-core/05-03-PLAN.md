## Objective
Build the backend orchestration layer that connects the analysis engine (Plan 05-01) and instruction applier (Plan 05-02) into a complete wizard pipeline. This includes: session state management for wizard checkpoints, the LLM Pass 2 prompt for generating insertion instructions, the iterative chat feedback endpoint using SSE streaming, the preview pipeline (render with GW dummy data -> PDF), and the download endpoint. Each wizard step has a REST endpoint, and state persists across page navigation.

## Tasks

### Task 1: Wizard Session State Manager
**Commit:** `feat(adapter): implement Redis-backed wizard session state management`
**Files:**
- backend/src/services/wizardState.ts -- new

**Steps:**
1. Create `backend/src/services/wizardState.ts`:
   - `WizardStep`: type union "upload" | "analysis" | "adaptation" | "preview" | "download"
   - `WizardState` interface:
     - `sessionId: string` (unique per wizard session)
     - `userId: string`
     - `currentStep: WizardStep`
     - `templateFile: { originalName: string; storagePath: string; base64: string; uploadedAt: string }`
     - `config: { templateType: string; language: string }`
     - `analysis: { mappingPlan: MappingPlan | null; referenceTemplateHash: string | null; llmPrompt: string | null }`
     - `adaptation: { instructions: InstructionSet | null; appliedDocxPath: string | null; appliedCount: number; skippedCount: number }`
     - `preview: { pdfJobId: string | null; pdfUrl: string | null; docxUrl: string | null }`
     - `chat: { iterationCount: number; history: Array<{ role: string; content: string; timestamp: string }> }`
     - `createdAt: string`
     - `updatedAt: string`
   - Redis key pattern: `layer8:wizard:{userId}:{sessionId}`
   - `createWizardSession(userId: string): Promise<WizardState>` -- Creates new state with unique sessionId, stores in Redis with 24h TTL
   - `getWizardSession(userId: string, sessionId: string): Promise<WizardState | null>` -- Reads from Redis
   - `updateWizardSession(userId: string, sessionId: string, updates: Partial<WizardState>): Promise<WizardState>` -- Merges updates, resets TTL
   - `deleteWizardSession(userId: string, sessionId: string): Promise<void>` -- Cleanup
   - `getActiveWizardSession(userId: string): Promise<WizardState | null>` -- Finds user's most recent active session by scanning keys
   - All reads/writes through `redisClient` (already configured in `@/db/redis.ts`)

### Task 2: Insertion Prompt & Apply Pipeline (Python)
**Commit:** `feat(adapter): add insertion prompt builder and apply endpoint`
**Files:**
- sanitization-service/app/services/insertion_prompt.py -- new
- sanitization-service/app/routes/adapter.py -- extend with /apply and /enrich endpoints

**Steps:**
1. Create `sanitization-service/app/services/insertion_prompt.py`:
   - `build_insertion_prompt(doc_structure: DocxStructure, mapping_plan: MappingPlan) -> str`:
     - System prompt section: Role as a DOCX template engineer, task is to generate precise insertion instructions
     - Section 1: Client document structure (numbered paragraphs with full text, tables with cell content)
     - Section 2: Approved mapping plan (each entry with section_index, gw_field, confidence)
     - Section 3: Output format -- JSON array of Instruction objects matching the InstructionSet schema
     - Section 4: Rules:
       - For each mapping entry, determine the exact text in the paragraph to replace
       - Use the original_text field to specify what to find (exact substring match)
       - Use replacement_text for the Jinja2 expression
       - For table rows: use wrap_table_row action to add loop markers
       - Preserve surrounding text -- only replace the specific content that maps to a GW field
       - If a paragraph contains multiple mappable fields, generate separate instructions for each
   - `build_insertion_system_prompt() -> str`: System prompt for Pass 2
2. Add to `sanitization-service/app/routes/adapter.py`:
   - `POST /apply`: Accepts `ApplyRequest` (template_base64 + instruction_set)
     - Validate instructions via jinja2_validator
     - Enrich instructions via rules_engine
     - Apply instructions via InstructionApplier
     - Return ApplyResponse with modified DOCX as base64 + counts
   - `POST /enrich`: Accepts InstructionSet, returns enriched InstructionSet (rules engine only, no DOCX modification)
   - `POST /build-insertion-prompt`: Accepts doc_structure + mapping_plan, returns prompt string

### Task 3: Backend Orchestration Endpoints (Steps 1-3)
**Commit:** `feat(adapter): add upload, analyze, and apply orchestration endpoints`
**Files:**
- backend/src/services/templateAdapter.ts -- extend
- backend/src/routes/templateAdapter.ts -- extend

**Steps:**
1. Extend `backend/src/services/templateAdapter.ts`:
   - `uploadTemplate(file: Express.Multer.File, templateType: string, language: string, userId: string): Promise<WizardState>`:
     - Create wizard session with createWizardSession()
     - Store file metadata and base64 in session state
     - Update currentStep to "upload"
     - Return session state
   - `applyInstructions(wizardState: WizardState): Promise<WizardState>`:
     - Get insertion prompt from Python service (/build-insertion-prompt)
     - Call LLM with insertion prompt (non-streaming for structured JSON)
     - Parse LLM response as instruction set
     - POST to Python service /apply with template + instructions
     - Save applied DOCX to uploads/documents/
     - Update wizard state with results
     - Log audit event
   - Error handling: On LLM failure, preserve last good state for retry
2. Extend `backend/src/routes/templateAdapter.ts`:
   - `POST /api/adapter/upload`:
     - Requires auth, multipart upload (file + templateType + language)
     - Calls uploadTemplate()
     - Returns `{ sessionId, currentStep: "upload" }`
   - `POST /api/adapter/apply`:
     - Requires auth, body: `{ sessionId }`
     - Loads wizard state, validates currentStep is "analysis"
     - Calls applyInstructions()
     - Returns `{ currentStep: "adaptation", appliedCount, skippedCount, warnings }`
   - All endpoints validate sessionId belongs to the authenticated user

### Task 4: Preview, Download & Chat Endpoints (Steps 4-5)
**Commit:** `feat(adapter): add preview, download, and iterative chat endpoints`
**Files:**
- backend/src/services/templateAdapter.ts -- extend
- backend/src/routes/templateAdapter.ts -- extend

**Steps:**
1. Extend `backend/src/services/templateAdapter.ts`:
   - `generatePreview(wizardState: WizardState): Promise<WizardState>`:
     - Read the applied DOCX from disk
     - Render with GW dummy report data (Report ID from config, default 1):
       - Read applied DOCX, encode as base64
       - Fetch GW data via fetchReportData(), map via mapReportToTemplateContext()
       - POST to sanitization-service /render-template
     - Save rendered DOCX and queue PDF conversion via addPdfConversionJob()
     - Update wizard state with pdfJobId and docxUrl
   - `getDownloadPath(wizardState: WizardState): string`:
     - Return the path to the applied DOCX (the one with Jinja2 placeholders, NOT the rendered one)
     - This is what the user downloads for upload to Ghostwriter
   - `processChatFeedback(wizardState: WizardState, userMessage: string): AsyncGenerator<LLMStreamChunk>`:
     - Build chat context: current mapping plan + user message + chat history
     - Stream LLM response via generateStream() with feature: 'template-adapter'
     - Increment chat.iterationCount in session state
     - If iterationCount >= 5, include soft warning in system prompt
     - Parse LLM response for mapping plan modifications (JSON blocks in response)
     - Update mapping plan in session state if modifications found
     - Log chat interaction to audit
2. Extend `backend/src/routes/templateAdapter.ts`:
   - `POST /api/adapter/preview`:
     - Requires auth, body: `{ sessionId }`
     - Loads state, validates currentStep is "adaptation"
     - Calls generatePreview()
     - Returns `{ pdfJobId, docxUrl }` with 202 Accepted
   - `GET /api/adapter/preview/:sessionId`:
     - Returns current preview status (polls pdfJobId)
   - `GET /api/adapter/download/:sessionId`:
     - Validates session belongs to user
     - Streams the applied DOCX file (with Jinja2 placeholders)
     - Logs audit event for download
   - `POST /api/adapter/chat`:
     - Requires auth, body: `{ sessionId, message }`
     - Sets SSE headers
     - Streams LLM response via processChatFeedback()
     - SSE events: delta (text chunk), mapping_update (modified mapping plan), done (usage), error
   - `GET /api/adapter/session/:sessionId`:
     - Returns full wizard state (for page reload/navigation restoration)
   - `GET /api/adapter/session`:
     - Returns user's active session if any (for sidebar badge or auto-resume)
