## Objective
Build the backend services for LLM Pass 1: analyzing client DOCX templates against reference templates to produce a field mapping plan. The reference template loader reads the matching GW template (by type and language), extracts all Jinja2 placeholder patterns, and feeds them alongside the parsed DOCX structure JSON to the LLM. The LLM returns a structured mapping plan (which client template sections map to which GW fields). This plan includes the analysis prompt engineering, data structures, and the backend API endpoint. All shared Pydantic models for the adapter feature (both analysis and instruction-related) are defined here so Plan 05-02 can run in parallel without file conflicts.

### Task 1: Adapter Pydantic Models (All) & Reference Template Loader
**Commit:** `feat(adapter): add all adapter Pydantic models, reference template loader, and pattern extractor`
**Files:**
- sanitization-service/app/models/adapter.py -- new (ALL adapter models: analysis + instruction)
- sanitization-service/app/services/reference_loader.py -- new

**Steps:**
1. Create `sanitization-service/app/models/adapter.py` with ALL Pydantic models for the adapter feature:
   **Analysis models (used by this plan):**
   - `TemplateType`: Literal["web", "internal", "mobile"]
   - `TemplateLanguage`: Literal["en", "pt-pt"]
   - `Jinja2Pattern`: pattern (str), marker_type (Literal["text", "paragraph_rt", "run_rt", "table_row_loop", "control_flow"]), gw_field (str, the target GW field path like "client.short_name"), context (str, surrounding text for LLM reference)
   - `ReferenceTemplateInfo`: template_type, language, filename, patterns (list[Jinja2Pattern]), placeholder_count (int)
   - `MappingEntry`: section_index (int, paragraph index in client doc), section_text (str, first 100 chars), gw_field (str), placeholder_template (str, the Jinja2 expression to insert), confidence (float 0-1), marker_type (str), rationale (str)
   - `MappingPlan`: entries (list[MappingEntry]), template_type (TemplateType), language (TemplateLanguage), warnings (list[str])
   - `AnalyzeRequest`: template_base64 (str), template_type (TemplateType), language (TemplateLanguage)
   - `AnalyzeResponse`: mapping_plan (MappingPlan), reference_template_hash (str)
   **Instruction models (consumed by Plan 05-02):**
   - `InstructionAction`: Literal["replace_text", "insert_before", "insert_after", "wrap_table_row"]
   - `Instruction`: action (InstructionAction), paragraph_index (int), original_text (str), replacement_text (str), marker_type (str), gw_field (str)
   - `InstructionSet`: instructions (list[Instruction]), template_type (TemplateType), language (TemplateLanguage), additional_blocks (list[str])
   - `ApplyRequest`: template_base64 (str), instruction_set (InstructionSet)
   - `ApplyResponse`: output_base64 (str), applied_count (int), skipped_count (int), warnings (list[str])
   - `ValidationResult`: valid (bool), errors (list[str]), sanitized_instructions (InstructionSet | None)
   **Constants (consumed by Plan 05-02):**
   - `FIELD_MARKER_MAP`: dict mapping GW field paths to required marker types (client.short_name -> "text", finding.description_rt -> "paragraph_rt", finding.severity_rt -> "run_rt", scope -> "table_row_loop", etc.)
   - `TEMPLATE_TYPE_FEATURES`: dict mapping template types to additional features (internal -> ["filter_type", "namespace_counters"], web -> ["scope_loops", "affected_entities"], mobile -> ["scope_loops", "affected_entities"])
2. Create `sanitization-service/app/services/reference_loader.py`:
   - `TEMPLATE_DIR` = path relative to project root: `test-templates/ghost-templates/`
   - `TEMPLATE_MAP`: dict mapping (type, language) -> filename. Cover all 8 templates. For pt-pt, use "O Cliente" variants as default.
   - `load_reference_template(template_type, language) -> ReferenceTemplateInfo`: Reads the matching DOCX file, calls pattern extractor, returns info with SHA-256 hash
   - `extract_jinja2_patterns(file_bytes: bytes) -> list[Jinja2Pattern]`: Uses DocxParserService to parse the reference template, then regex-scans all paragraph text, table cell text, header/footer text for Jinja2 patterns:
     - `{{p\s+(\S+)\s*}}` -- paragraph rich text markers
     - `{{r\s+(\S+)\s*}}` -- run rich text markers
     - `{{\s*([^}]+?)\s*}}` -- simple text placeholders (excluding p/r variants)
     - `{%tr\s+(.+?)\s*%}` -- table row loop markers
     - `{%\s+(.+?)\s*%}` -- other control flow (set namespace, endfor, etc.)
   - Each extracted pattern includes the surrounding paragraph text as context
   - Map extracted placeholder names to GW field paths using a static lookup dict derived from the README catalogue

### Task 2: Analysis Prompt Engineering (Python)

- sanitization-service/app/services/analysis_prompt.py -- new

**Steps:**
1. Create `sanitization-service/app/services/analysis_prompt.py`:
   - `build_analysis_prompt(doc_structure: DocxStructure, reference_info: ReferenceTemplateInfo, template_type: TemplateType, language: TemplateLanguage) -> str`:
     - System prompt section: Role as document analysis expert, task description (map client template sections to GW fields)
     - Section 1: Condensed representation of client DOCX structure -- numbered paragraphs with text (truncated to 200 chars), heading levels, table locations. Only include non-empty paragraphs.
     - Section 2: Reference template patterns -- all extracted Jinja2 patterns with their marker types and GW field mappings
     - Section 3: Available GW fields with descriptions (derived from TemplateContext interface)
     - Section 4: Output format specification -- JSON array of MappingEntry objects. Explicit JSON schema with examples.
     - Section 5: Rules -- only map sections that semantically match a GW field, prefer high confidence matches, flag ambiguous sections in warnings, do NOT map boilerplate/legal text
   - `build_analysis_system_prompt() -> str`: System prompt establishing the LLM role as a document structure analyst for pentest report templates
   - Keep prompts concise to minimize token usage (Sonnet 4.5 for template-adapter feature)

### Task 3: Analysis API Route (Python)
**Commit:** `feat(adapter): add POST /analyze endpoint for template analysis`
**Files:**
- sanitization-service/app/routes/adapter.py -- new
- sanitization-service/app/routes/__init__.py -- update
- sanitization-service/app/main.py -- mount new router

**Steps:**
1. Create `sanitization-service/app/routes/adapter.py` with APIRouter:
   - `POST /analyze`: Accepts `AnalyzeRequest` JSON body
     - Decode base64 template to bytes, validate as DOCX
     - Parse the client DOCX using DocxParserService
     - Load the matching reference template using reference_loader
     - Build the analysis prompt
     - Return the prompt, parsed structure, and reference info as response (actual LLM call is made by Node.js backend which owns the LLM client)
     - Response model: `AnalyzeResponse` containing the prepared prompt, doc_structure summary, and reference_template_hash
   - Note: The Python service does NOT call the LLM directly. It prepares the prompt and returns it. The Node.js backend calls the LLM and sends the response back for validation.
   - `POST /validate-mapping`: Accepts raw LLM response text + AnalyzeRequest context
     - Parses LLM JSON response into MappingPlan
     - Validates all entries: section_index within range, gw_field is valid, marker_type is appropriate for the field
     - Returns validated MappingPlan or error details
2. Update `app/routes/__init__.py` to export `adapter_router`
3. Update `app/main.py` to mount: `app.include_router(adapter_router, prefix="/adapter", tags=["adapter"])`

### Task 4: Backend Analysis Orchestration (Node.js)
**Commit:** `feat(adapter): add backend analysis orchestration with LLM integration`
**Files:**
- backend/src/services/templateAdapter.ts -- new
- backend/src/routes/templateAdapter.ts -- new
- backend/src/index.ts -- mount new router

**Steps:**
1. Create `backend/src/services/templateAdapter.ts`:
   - `analyzeTemplate(templateBase64: string, templateType: string, language: string): Promise<AnalysisResult>`:
     - POST to sanitization-service `/adapter/analyze` to get prepared prompt and reference info
     - Call LLM with the analysis prompt using `createLLMClient()` with `feature: 'template-adapter'`
     - Collect full LLM response (non-streaming for structured JSON output)
     - POST to sanitization-service `/adapter/validate-mapping` with LLM response for validation
     - Return validated mapping plan + reference template hash
   - TypeScript interfaces: `AnalysisResult { mappingPlan: MappingPlan; referenceTemplateHash: string }`, `MappingPlan { entries: MappingEntry[]; templateType: string; language: string; warnings: string[] }`, `MappingEntry { sectionIndex: number; sectionText: string; gwField: string; placeholderTemplate: string; confidence: number; markerType: string; rationale: string }`
2. Create `backend/src/routes/templateAdapter.ts`:
   - `POST /api/adapter/analyze`:
     - Requires auth, accepts multipart (file + type + language fields) via multer
     - Validates file is .docx, type is web|internal|mobile, language is en|pt-pt (Zod)
     - Reads file, converts to base64, calls `analyzeTemplate()`
     - Logs audit event with reference_template_hash
     - Returns mapping plan JSON
   - Mount in `backend/src/index.ts`: `app.use('/api/adapter', templateAdapterRouter)`
