## Objective
Build the integration and end-to-end test suite that proves the complete template adapter pipeline works correctly for all three template types. Tests cover the Python services (instruction applier, rules engine, validator), the backend orchestration endpoints, and the full end-to-end flow from DOCX upload to GW-compatible output. The E2E test uses actual reference templates and static GW fixture data to verify the output is both valid DOCX and renders correctly through the existing Jinja2 rendering pipeline.

### Task 1: Python Pipeline Integration Tests
**Commit:** `test(adapter): add Python pipeline integration tests for all template types`
**Files:**
- sanitization-service/tests/test_adapter_pipeline.py -- new

**Steps:**
1. Create `sanitization-service/tests/test_adapter_pipeline.py`:
   - Fixture: Create a simple client DOCX programmatically with python-docx containing:
     - Heading "Executive Summary" with bold formatting
     - Paragraph "Client Name: ACME Corp" with specific font (Calibri 11pt)
     - Table with headers ["Finding", "Severity", "Description"] and one data row
     - Paragraph "Project dates: January 2026 - March 2026"
     - Paragraph "Testing scope: www.example.com"
   - `test_pipeline_web_template()`:
     - Build a mock mapping plan for web template type:
       - "ACME Corp" -> `{{ client.short_name }}` (text marker)
       - "Finding"/"Severity"/"Description" table row -> findings loop with `{%tr%}` markers
       - "January 2026 - March 2026" -> `{{ project.start_date }} - {{ project.end_date }}`
       - "www.example.com" -> scope loop
     - Build instruction set from mapping plan
     - Enrich via rules_engine (should add `{{p}}` markers for rich text, `{%tr%}` for loops)
     - Validate via jinja2_validator (should pass)
     - Apply via InstructionApplier
     - Verify output DOCX:
       - Contains `{{ client.short_name }}` in correct paragraph
       - Contains `{%tr for finding in findings %}` in table
       - Original font (Calibri 11pt) preserved on modified paragraphs
       - Heading style preserved
   - `test_pipeline_internal_template()`:
     - Similar but with filter_type() features
     - Verify `filter_type(["Web"])` injected for categorised findings
     - Verify `{% set ns = namespace(counter=0) %}` injected for numbering
   - `test_pipeline_mobile_template()`:
     - Similar to web with scope loops and affected_entities
   - `test_pipeline_invalid_instructions_rejected()`:
     - Build instructions with unsafe Jinja2 (e.g., `{{ ''.__class__.__mro__ }}`)
     - Verify validator rejects them
     - Verify applier is never called
   - `test_pipeline_preserves_formatting()`:
     - Create DOCX with specific formatting: red bold text, 14pt heading, blue italic
     - Apply instructions that replace text within formatted runs
     - Re-parse output DOCX and verify formatting attributes unchanged

### Task 4: E2E Template Adaptation Test
**Commit:** `test(adapter): add end-to-end template adaptation verification`
**Files:**
- sanitization-service/tests/test_adapter_e2e.py -- new

**Steps:**
1. Create `sanitization-service/tests/test_adapter_e2e.py` (marked `@pytest.mark.integration`):
   - This test uses actual reference templates as both the "client template" (simulating a client's template that has the same structure but different text) and as the reference
   - `test_e2e_web_en_adaptation()`:
     - Load Web EN reference template from `test-templates/ghost-templates/`
     - Parse it via DocxParserService to get structure
     - Load reference template info via reference_loader
     - Build a synthetic mapping plan that maps known sections to GW fields:
       - Find paragraphs containing "Client Name" -> map to client.short_name
       - Find findings table -> map to findings loop
       - Find scope table -> map to scope loop
     - Build instruction set from mapping
     - Enrich instructions via rules_engine
     - Apply instructions via InstructionApplier
     - Verify output DOCX:
       - Is valid (parseable by python-docx)
       - Contains expected Jinja2 placeholders
       - Preserves original page layout and styles
     - **Critical verification**: Render the adapted DOCX through TemplateRendererService with GW fixture data
       - This proves the output is GW-compatible
       - Rendered output should contain actual GW data values (e.g., "AI Template Engine" for client name)
   - `test_e2e_internal_en_adaptation()`:
     - Same flow with Internal EN template
     - Verify filter_type() and namespace counters present in output
     - Verify render with GW fixture data succeeds
   - `test_e2e_mobile_en_adaptation()`:
     - Same flow with Mobile EN template
     - Verify scope loops and affected_entities markers present
     - Verify render with GW fixture data succeeds
2. Note: These tests do NOT require a running LLM (mapping plan is synthetic) or Ghostwriter instance (uses static fixture data)
