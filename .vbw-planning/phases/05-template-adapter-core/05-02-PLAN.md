## Objective
Build the Python services for LLM Pass 2: taking a validated mapping plan and applying Jinja2 placeholders to the client DOCX. The instruction applier receives structured JSON instructions (from the LLM) and modifies the DOCX via python-docx while preserving all original formatting. The rules engine auto-applies rich text markers and template-type-specific Jinja2 features. The Jinja2 validator ensures only safe, whitelisted syntax is inserted. This plan does NOT involve the LLM prompt for Pass 2 (that is in Plan 05-03 orchestration). All Pydantic models and constants (Instruction, InstructionSet, FIELD_MARKER_MAP, TEMPLATE_TYPE_FEATURES) are defined in Plan 05-01's adapter.py -- this plan only imports from that module.

**File coordination note:** This plan imports models from `sanitization-service/app/models/adapter.py` which is created by Plan 05-01. Both plans are wave 1 but have no file write conflicts -- this plan only creates NEW service files and test files, never writes to adapter.py.

## Tasks

### Task 1: Jinja2 Syntax Validator
**Commit:** `feat(adapter): implement Jinja2 syntax whitelist validator`
**Files:**
- sanitization-service/app/services/jinja2_validator.py -- new

**Steps:**
1. Create `sanitization-service/app/services/jinja2_validator.py`:
   - Imports from `app.models.adapter`: Instruction, InstructionSet, ValidationResult
   - `ALLOWED_VARIABLES`: Set of allowed Jinja2 variable paths -- all known GW field paths from TemplateContext (client.short_name, project.start_date, etc.), loop variables (finding.*, item.*, team[0].*, loop.index), and computed fields (totals.findings)
   - `ALLOWED_FILTERS`: Set of allowed Jinja2 filters -- filter_type, default, int, string, lower, upper, title (safe subset)
   - `ALLOWED_CONTROL`: Allowed control flow patterns -- "for ... in findings", "for ... in scope", "for ... in team", "set ns = namespace", "endfor"
   - `validate_instruction(instruction: Instruction) -> list[str]`: Validates a single instruction:
     - Parse the replacement_text for Jinja2 syntax
     - Check all variable references against ALLOWED_VARIABLES
     - Check all filter usage against ALLOWED_FILTERS
     - Check control flow against ALLOWED_CONTROL patterns
     - Return list of error messages (empty = valid)
   - `validate_instruction_set(instruction_set: InstructionSet) -> ValidationResult`:
     - Validate each instruction
     - Check paragraph_index is non-negative
     - Check no duplicate paragraph_index + action combinations
     - Return ValidationResult with sanitized instructions (those that passed) or errors
   - `_parse_jinja2_expression(expr: str) -> dict`: Simple parser that extracts variable names, filter names, and control keywords from a Jinja2 expression string. Uses regex, not a full Jinja2 AST parser.

### Task 2: Rules Engine (Marker & Feature Injection)
**Commit:** `feat(adapter): implement template-type-aware rules engine for marker injection`
**Files:**
- sanitization-service/app/services/rules_engine.py -- new

**Steps:**
1. Create `sanitization-service/app/services/rules_engine.py`:
   - Imports from `app.models.adapter`: InstructionSet, FIELD_MARKER_MAP, TEMPLATE_TYPE_FEATURES
   - `apply_marker_rules(instructions: InstructionSet) -> InstructionSet`:
     - For each instruction, check the gw_field against FIELD_MARKER_MAP
     - If the LLM specified a plain `{{ field }}` but the field requires `{{p field }}` or `{{r field }}`, rewrite the replacement_text with the correct marker
     - Rich text fields (those ending in _rt) always get paragraph-level markers: `{{p finding.description_rt }}`
     - Severity uses run-level: `{{r finding.severity_rt }}`
     - Table row loops get `{%tr for finding in findings %}` / `{%tr endfor %}`
     - Return the modified instruction set
   - `inject_type_features(instructions: InstructionSet) -> InstructionSet`:
     - Based on template_type, add required additional_blocks:
     - For "internal": Add `{% set ns = namespace(counter=0) %}` before findings section, add namespace counter increments within each category loop
     - For "web"/"mobile": Add `{%tr for item in scope %}` block for scope table if scope field is present in mappings
     - Add `{{ '%02d' % loop.index }}` for finding numbering in loops
   - `enrich_instructions(instructions: InstructionSet) -> InstructionSet`:
     - Pipeline: apply_marker_rules -> inject_type_features -> return
     - Main entry point called by the orchestration layer

### Task 3: Instruction Applier (DOCX Modifier)
**Commit:** `feat(adapter): implement DOCX instruction applier preserving formatting`
**Files:**
- sanitization-service/app/services/instruction_applier.py -- new

**Steps:**
1. Create `sanitization-service/app/services/instruction_applier.py`:
   - Imports from `app.models.adapter`: InstructionSet, Instruction
   - `class InstructionApplier`:
     - `apply(template_bytes: bytes, instruction_set: InstructionSet) -> tuple[bytes, int, int, list[str]]`:
       - Load DOCX via `Document(BytesIO(template_bytes))`
       - Sort instructions by paragraph_index descending (apply from bottom up to avoid index shifts)
       - For each instruction:
         - `replace_text`: Find the run(s) in the target paragraph containing original_text, replace with replacement_text while preserving the run's formatting (bold, italic, font, size, color). If original_text spans multiple runs, merge them into one run with the first run's formatting.
         - `insert_before`: Insert a new paragraph before the target with the replacement_text, copying the target paragraph's style
         - `insert_after`: Insert a new paragraph after the target with the replacement_text, copying the target paragraph's style
         - `wrap_table_row`: Find the table row containing the target paragraph, insert loop markers (`{%tr for ... %}` before row, `{%tr endfor %}` after row). This requires XML-level manipulation of the table element.
       - Track applied count, skipped count, and warnings for each instruction
       - Save modified document to BytesIO and return bytes
     - `_replace_in_paragraph(paragraph, original_text: str, replacement_text: str) -> bool`:
       - Iterate runs, find the run(s) containing original_text
       - Strategy: If original_text is entirely within one run, replace in that run (preserving formatting). If it spans multiple runs, consolidate into the first matching run and remove the others.
       - Return True if replacement was made, False otherwise
     - `_preserve_formatting(source_run, target_run)`:
       - Copy font properties: name, size, bold, italic, underline, color, strike
       - Copy paragraph properties if needed: alignment, indentation
     - `_wrap_table_row_with_loop(doc, paragraph_index: int, loop_expr: str) -> bool`:
       - Navigate from paragraph to its parent table row via XML tree
       - Insert `{%tr` loop start before the row and `{%tr endfor %}` after the row in the table XML
       - Uses `lxml.etree` operations on the underlying OOXML
2. Handle edge cases:
   - Paragraph text that has been split across many runs (common in DOCX from Word)
   - Table cells with nested paragraphs
   - Headers/footers (skip for v1, only modify body paragraphs)
