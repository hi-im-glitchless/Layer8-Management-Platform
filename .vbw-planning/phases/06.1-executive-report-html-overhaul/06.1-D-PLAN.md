## Objective

Overhaul the LLM generation pipeline to produce styled HTML instead of plain text, replace matplotlib chart rendering with Chart.js JSON config computation, and rewrite the Python report builder to assemble HTML from the skeleton template. This transforms Pass 2 from "LLM generates plain text -> Python fills DOCX skeleton" to "LLM generates HTML with CSS classes -> Python assembles complete HTML document."

## Context

- `@sanitization-service/app/services/report_narrative_prompt.py` -- current plain-text prompt to modify for HTML output
- `@sanitization-service/app/services/chart_renderer.py` -- current matplotlib renderer to replace with Chart.js config computation
- `@sanitization-service/app/services/report_builder.py` -- current DOCX builder to rewrite as HTML assembler
- `@sanitization-service/app/routes/report.py` -- endpoints to modify (render-charts -> compute-chart-data, build-report -> build-html-report)
- `@sanitization-service/app/models/report.py` -- Pydantic models to update
- `@backend/src/services/reportService.ts` -- generation pipeline to update for HTML flow
- `@test-templates/executive/report-template.css` -- CSS classes the LLM must use (from Plan 06.1-A)
- `@test-templates/executive/skeleton-en.html` -- skeleton structure the LLM follows (from Plan 06.1-A)
- `@06.1-CONTEXT.md` decisions: #5 LLM receives structured data + HTML skeleton reference, #6 LLM outputs complete HTML, #7 Chart.js for charts, #8 frontend renders Chart.js, #10 static CSS + LLM uses CSS classes, #13 keep extraction+metrics rewrite builder+charts, #16 backend produces HTML with chart configs

### Task 2: Replace matplotlib chart rendering with Chart.js config computation

**Files:**
- `sanitization-service/app/services/chart_data.py` (new)
- `sanitization-service/app/services/chart_renderer.py` (delete)
- `sanitization-service/app/models/report.py` (modify)
- `sanitization-service/app/routes/report.py` (modify)

**What:** Replace the matplotlib-based chart renderer with a pure data computation module that produces Chart.js JSON configurations. Charts are no longer rendered server-side -- they are rendered by Chart.js in the frontend iframe and by Gotenberg's Chromium during PDF conversion.

New `chart_data.py` module:
- `compute_chart_configs(severity_counts, category_counts, stacked_data, compliance_scores, risk_score) -> dict[str, dict]`
- Returns a dict mapping chart ID to Chart.js config object:
  - `severity_pie`: `{ type: 'pie', data: { labels, datasets }, options }`
  - `category_bar`: `{ type: 'bar', data: { ... }, options: { indexAxis: 'y' } }`
  - `stacked_severity`: `{ type: 'bar', data: { ... }, options: { scales: { x: { stacked: true }, y: { stacked: true } } } }`
  - `compliance_radar`: `{ type: 'radar', data: { ... } }`
  - `risk_score`: `{ type: 'doughnut', data: { ... } }` (donut gauge)
  - `top_vulnerabilities`: `{ type: 'bar', data: { ... } }`
- All configs use colors from `report_theme.py` (SEVERITY_COLORS, BRAND_COLORS)
- Export `CSS_COLORS` dict from report_theme.py for use in Chart.js configs

Update routes:
- Rename endpoint `POST /report/render-charts` to `POST /report/compute-chart-data`
- Update Pydantic models: `RenderChartsRequest` -> `ComputeChartDataRequest`, `RenderChartsResponse` -> `ComputeChartDataResponse` (values are dicts not base64 strings)

Delete `chart_renderer.py` (matplotlib) entirely.

**Acceptance:**
- [ ] `chart_data.py` produces valid Chart.js JSON configs for all 6 chart types
- [ ] Configs use colors from report_theme.py
- [ ] `POST /report/compute-chart-data` endpoint returns dict of chart configs
- [ ] `chart_renderer.py` deleted
- [ ] No matplotlib imports remain in report-related code
- [ ] Pydantic models updated

**Commit:** `feat(report): replace matplotlib charts with Chart.js config computation`

### Task 3: Rewrite report_builder.py as HTML assembler

**Files:**
- `sanitization-service/app/services/report_builder.py` (rewrite)

**What:** Rewrite the report builder from a python-docx DOCX filler to an HTML assembler. Instead of filling a skeleton DOCX with text and chart images, it now reads an HTML skeleton, injects narrative sections into section markers, embeds Chart.js configs as `data-config` attributes on canvas elements, fills cover page metadata, and returns a complete HTML document.

New `ReportBuilder` class:
- `__init__(self, skeleton_path: str)` -- reads the HTML skeleton file
- `build_report(self, report_data: dict, chart_configs: dict) -> str` -- returns complete HTML string

`build_report()` flow:
1. Read skeleton HTML from file
2. Inline the CSS: read `report-template.css` and inject into `<style>` tag in `<head>` (for Gotenberg compatibility)
3. Fill cover page metadata: replace `[CLIENT_NAME]`, `[PROJECT_CODE]`, `[REPORT_DATE]`, `[START_DATE]`, `[END_DATE]` placeholders
4. Fill section content: for each `<section data-section="executive_summary">`, find the `<div class="section-content">` inside and replace its contents with the LLM-generated HTML
5. Embed chart configs: for each `<canvas data-chart="severity_pie">`, set `data-config='${json.dumps(config)}'` attribute
6. Bundle Chart.js: ensure the Chart.js CDN script and initialization script are present

Update `get_skeleton_path()` to look for `.html` files instead of `.docx`:
- `skeleton-en.html` and `skeleton-pt-pt.html` at `test-templates/executive/`

Update Pydantic models:
- `BuildReportRequest.chart_images` -> `BuildReportRequest.chart_configs` (dict of dicts instead of dict of base64 strings)
- `BuildReportResponse.docx_base64` -> `BuildReportResponse.html_content` (HTML string instead of base64 DOCX)
- Remove `filename` from response (no file, just HTML string)

**Acceptance:**
- [ ] `build_report()` returns valid HTML string with inlined CSS
- [ ] Cover page metadata placeholders replaced
- [ ] Section content injected at correct `data-section` markers
- [ ] Chart.js configs embedded as `data-config` attributes on canvas elements
- [ ] Skeleton path resolution uses `.html` extension
- [ ] No python-docx imports remain in report_builder.py

**Commit:** `feat(report): rewrite report builder as HTML assembler from skeleton`

### Task 4: Update generation pipeline in reportService.ts for HTML

**Files:**
- `backend/src/services/reportService.ts` (modify)

**What:** Update the `generateReport()` function to produce HTML instead of DOCX. The pipeline flow changes:

Old: compute-metrics -> render-charts (PNG) -> narrative (plain text) -> desanitize -> build-report (DOCX) -> Gotenberg (DOCX->PDF)
New: compute-metrics -> compute-chart-data (JSON) -> narrative (HTML) -> build-html-report (HTML) -> store HTML in session -> Gotenberg (HTML->PDF)

Key changes to `generateReport()`:
1. Stage 1 (computing): Same -- call `/report/compute-metrics`
2. Stage 2 (chart_data): Call `/report/compute-chart-data` instead of `/report/render-charts`, get Chart.js JSON configs instead of base64 PNGs
3. Stage 3 (narrative): Call LLM Pass 2 with updated prompt, get HTML sections instead of plain text
4. Stage 4 (assembling_html): Call `/report/build-html-report` instead of `/report/build-report`, send narrative HTML sections + chart configs, get complete HTML string back
5. Stage 5 (converting_pdf): Send HTML to Gotenberg `POST /forms/chromium/convert/html` instead of LibreOffice conversion. Include waitDelay parameter for Chart.js rendering.
6. Store `generatedHtml` and `chartConfigs` in session instead of `reportDocxPath`

De-sanitization change: Do NOT de-sanitize before building the HTML. The generated HTML stays sanitized. De-sanitization is a frontend-only toggle in the Review step (decision #11). Remove the pre-build desanitize loop.

SSE stage events update: `computing` -> `chart_data` -> `narrative` -> `assembling_html` -> `converting_pdf`

Also update `processReportChat()`: the section correction flow now regenerates HTML for a single section, updates the generatedHtml in session, and re-submits to Gotenberg for PDF.

**Acceptance:**
- [ ] `generateReport()` calls compute-chart-data (not render-charts)
- [ ] LLM Pass 2 produces HTML sections
- [ ] Report builder returns HTML string (not DOCX base64)
- [ ] Gotenberg receives HTML via `/forms/chromium/convert/html`
- [ ] `generatedHtml` and `chartConfigs` stored in session
- [ ] No `reportDocxPath` written to session
- [ ] No de-sanitization before HTML assembly (frontend-only)
- [ ] SSE stages reflect new pipeline
- [ ] `processReportChat()` updates HTML section and re-converts PDF

**Commit:** `feat(report): update generation pipeline for HTML output and Gotenberg HTML input`
