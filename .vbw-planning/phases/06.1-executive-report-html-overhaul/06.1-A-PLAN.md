## Objective

Create the HTML skeleton templates that define the Template Executivo layout, the CSS stylesheet for PDF rendering, and the mammoth.js DOCX-to-HTML conversion utility. These are the foundation artifacts that all subsequent plans depend on -- the skeleton defines what the LLM generates, the CSS defines how it looks, and the converter replaces the DOCX parsing path.

## Context

- `@test-templates/executive/Template Executivo.pdf` -- the visual reference for HTML skeleton design (cover page, red branding, Roboto fonts, metric cards, section structure, CONFIDENCIAL badge)
- `@sanitization-service/app/services/report_builder.py` -- current DOCX builder with section heading maps (EN + PT) to replicate in HTML
- `@sanitization-service/app/services/report_theme.py` -- SEVERITY_COLORS and BRAND_COLORS to translate to CSS custom properties
- `@06.1-CONTEXT.md` decisions: #1 mammoth.js, #17 mammoth body + docx_parser.py headers/footers, #20 single HTML per language with section markers, #21 Chart.js inline bundle + waitDelay, #22 Google Fonts CDN for Roboto, #23 CSS @page rules for pagination
- `@backend/package.json` -- add mammoth dependency here

### Task 4: Add mammoth.js dependency and create docxToHtml.ts utility

**Files:**
- `backend/package.json` (modify -- add mammoth dependency)
- `backend/src/services/docxToHtml.ts` (new)

**What:** Install mammoth.js (`mammoth` npm package) and create a utility module that converts DOCX buffer to clean HTML. This replaces the `docx_parser.py` paragraph extraction for the upload pipeline.

The `docxToHtml.ts` module exports:
- `convertDocxToHtml(buffer: Buffer): Promise<{ html: string; warnings: string[] }>` -- calls `mammoth.convertToHtml()` with style mappings that produce semantic HTML (h1-h6, p, table, ul/ol, img). Returns the HTML body content and any mammoth conversion warnings.
- Uses mammoth options: `styleMap` to map common DOCX styles to HTML elements, `convertImage` to inline base64 images.

**Acceptance:**
- [ ] `mammoth` listed in backend/package.json dependencies
- [ ] `docxToHtml.ts` exports `convertDocxToHtml` function
- [ ] Function accepts Buffer, returns `{ html, warnings }`
- [ ] TypeScript compiles without errors

**Commit:** `feat(report): add mammoth.js DOCX-to-HTML conversion utility`

### Task 5: Add docx_parser.py supplementary text extraction endpoint

**Files:**
- `sanitization-service/app/services/docx_parser.py` (modify)
- `sanitization-service/app/models/report.py` (modify -- add request/response models)
- `sanitization-service/app/routes/report.py` (modify -- add endpoint)

**What:** Add a new Python endpoint for extracting headers, footers, and text box content from DOCX files. mammoth.js handles the body, but these supplementary elements need python-docx extraction since mammoth doesn't parse them.

New endpoint: `POST /report/extract-supplementary` accepts `{ template_base64: str }` and returns `{ headers: string[], footers: string[], text_boxes: string[] }`.

Implementation in `docx_parser.py`: add `extract_supplementary_text(doc_bytes: bytes) -> dict` that uses python-docx to iterate `doc.sections` for headers/footers, and parses the XML for text box content (`<w:txbxContent>`).

**Acceptance:**
- [ ] New Pydantic models: `ExtractSupplementaryRequest`, `ExtractSupplementaryResponse`
- [ ] Endpoint `POST /report/extract-supplementary` returns headers, footers, text_boxes arrays
- [ ] Works with the test DOCX at `test-templates/executive/tech-report.docx`

**Commit:** `feat(report): add supplementary text extraction for headers/footers/text-boxes`
