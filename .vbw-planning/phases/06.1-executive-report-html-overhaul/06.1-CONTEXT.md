# Phase 6.1: Executive Report HTML Overhaul — Context

## User Vision

Replace the DOCX-centric executive report pipeline (built in Phase 6) with an HTML-centric approach. The uploaded pentest report is converted to HTML immediately, sanitized in HTML, previewed as interactive HTML with an editable mapping table, and the LLM generates a styled HTML report matching Template Executivo.pdf. The user reviews, corrects, de-sanitizes, and downloads as PDF only. No DOCX intermediary, no matplotlib charts — everything is HTML-native.

**Motivation:** Phase 6 UAT revealed fundamental UX issues: text diff view was wrong for sanitization review, deny list was confusing, IP addresses weren't incrementing, headers/footers weren't parsed, and react-pdf couldn't support text selection for entity mapping. HTML solves all of these natively.

## Essential Features

### DOCX→HTML Conversion
- **Library:** mammoth.js in the Node backend (purpose-built DOCX→HTML, handles styles/tables/images)
- **Headers/footers/text boxes:** mammoth handles body; docx_parser.py (Python) extracts headers, footers, and text box content as separate text segments for sanitization
- mammoth output is clean semantic HTML (headings, lists, tables, images)
- HTML becomes the source of truth from upload onwards

### Sanitization Pipeline
- Upload auto-triggers full pipeline: DOCX→HTML→sanitize→extract (same single-pipeline pattern as Phase 6)
- SSE progress events: 'uploading' → 'converting' → 'sanitizing' → 'extracting'
- **HTML text node sanitization:** Walk HTML DOM in Node, extract text nodes, batch-send to Presidio `/sanitize`
- **Entity span injection:** Replace sanitized text with `<span class="entity entity-person" data-entity-type="PERSON" data-placeholder="[PERSON_1]" data-original="John Doe">` tags
- **Incrementing placeholders:** Session-scoped counter map `Map<entityType, Map<originalValue, index>>` stored in Redis. Same value always gets same placeholder. New unique value increments the counter
- Headers/footers/text boxes sanitized alongside body text, stored as supplementary segments

### Sanitize & Review Step (Step 2)
- **HTML preview:** Sandboxed iframe with `srcdoc` attribute. CSS isolation from app UI. Chart.js can run inside (allow-scripts). Text selection events via postMessage to parent React app
- **Mapping table:** Toggleable right panel (full-width preview by default, 60/40 split when table visible)
  - 4 columns: Original Value | Placeholder | Entity Type (dropdown) | Actions (edit ✏️ / delete ✕)
  - Pre-filled with all Presidio-detected entities
  - Editing entity type auto-updates the placeholder
  - Deleting removes entity, restores original text in HTML
- **Text selection → entity creation:** Select text in iframe → floating popover appears → entity type dropdown (PERSON, ORG, IP_ADDRESS, etc.) → "Add Mapping" button → new row in mapping table, placeholder auto-generated with next increment
- **Entity types dropdown:** ~15 types — Presidio standard (PERSON, ORGANIZATION, LOCATION, EMAIL_ADDRESS, PHONE_NUMBER, IP_ADDRESS, URL, DATE_TIME, CREDIT_CARD) + pentest-specific (HOSTNAME, DOMAIN, MAC_ADDRESS, CVE_ID, NETWORK_RANGE) + CUSTOM
- **Deny list merged:** No separate deny list editor. Manual entity additions via the mapping table replace the deny list concept entirely
- **Metadata editor:** Collapsible accordion section below the HTML preview. Shows LLM-extracted values (may contain sanitized placeholders). Fields: client name, project code, start date, end date, scope summary

### LLM Generation Pipeline
- **Same 2-pass architecture as Phase 6:**
  - Pass 1 (Extract): Python builds prompt, Node calls Opus 4.6, Python validates → structured findings JSON + metadata
  - Python Compute: Risk score, compliance mapping, chart data as Chart.js JSON configs (not matplotlib PNGs)
  - Pass 2 (Generate): LLM receives structured data + HTML skeleton reference → outputs complete styled HTML document
- **LLM input:** Sanitized findings JSON, computed metrics/risk score, chart data (Chart.js format), metadata, HTML skeleton template as reference
- **LLM output:** Complete HTML document with all 12+ sections, CSS classes from report-template.css, chart `<canvas data-config>` placeholders with computed data

### HTML Skeleton Template
- **Single HTML file per language:** `skeleton-en.html`, `skeleton-pt-pt.html`
- Full Template Executivo design: cover page, section headers, metric card grid, chart containers, compliance tables, recommendation cards, footer
- Content areas marked with `<!-- SECTION: executive_summary -->` or `<div data-section="executive_summary"></div>` markers
- **Static CSS:** `report-template.css` defines all styling — colors (#ED1C24 brand), Roboto fonts (Google Fonts CDN), metric card grid, section layouts, severity color palette, CONFIDENCIAL badge
- **Page breaks:** CSS `@page` rules for A4 size/margins + `page-break-before: always` on major section headings
- **Fonts:** `<link>` to Google Fonts CDN (Roboto + Roboto Condensed) in HTML `<head>`. Gotenberg's Chromium loads them with internet access

### Chart.js Integration
- **Library:** Chart.js (canvas-based, ~60KB, professional defaults)
- **Chart types:** Pie (severity distribution), horizontal bar (categories), stacked bar (severity by category), radar (compliance), doughnut (risk score gauge), bar (top vulnerabilities)
- **Data flow:** Python computes chart data → JSON configs in Chart.js format → stored in session → HTML has `<canvas data-chart="severity_pie" data-config="...">` placeholders → frontend hydrates in iframe, Gotenberg executes bundled Chart.js during PDF conversion
- **Gotenberg handling:** Chart.js minified script bundled inline in HTML `<head>`. Initialization script at bottom reads data-config attributes and creates charts. `waitDelay` parameter ensures charts render before PDF capture

### Review Step (Step 4)
- **HTML preview:** Same sandboxed iframe, now showing LLM-generated styled report
- **De-sanitization toggle:** Button switches between sanitized view (default, what LLM produced) and de-sanitized view (real values). De-sanitization is frontend-only: find-replace placeholders with original values from mapping table. **LLM never sees de-sanitized values**
- **Chat corrections:** Same pattern as Phase 6 — user describes corrections, LLM regenerates affected HTML section in-place. Targeted section regeneration (only affected section, charts only if data changed)
- **Chat panel:** Same right-panel layout used in template adapter

### Download Step (Step 5)
- **PDF only** — no DOCX download (clients only accept PDF)
- HTML→PDF via Gotenberg HTML input endpoint (`/forms/chromium/convert/html`)
- Gotenberg's Chromium renders Chart.js charts, loads Google Fonts, respects @page CSS rules
- Download button for PDF file

## Technical Preferences

### Frontend Architecture
- **Same module path:** `features/executive-report/` (replace in-place, no v2 module)
- **KEEP:** ReportWizardShell (wizard navigation), StepUpload (upload + pipeline), ReportStepIndicator
- **REWRITE:** StepSanitizeReview (HTML preview + mapping table), StepReview (HTML preview + toggle + chat)
- **MODIFY:** StepGenerate (update progress stages for HTML pipeline), StepDownload (PDF-only)
- **New components:** EntityMappingTable, EntityPopover (text selection), HtmlReportPreview (iframe wrapper), DesanitizeToggle
- **iframe preview:** `<iframe srcdoc={html} sandbox="allow-scripts">` — CSS isolation, Chart.js runs inside, text selection via postMessage

### Backend Architecture
- **Routes:** Modify existing `executiveReport.ts` endpoints — upload returns HTML instead of storing DOCX, sanitize works on HTML text nodes, generate returns HTML instead of DOCX path
- **New utility:** `docxToHtml.ts` — mammoth.js wrapper for DOCX→HTML conversion
- **Session model updates:**
  - ADD: `uploadedHtml`, `sanitizedHtml`, `entityMappings`, `generatedHtml`, `chartConfigs`
  - REMOVE: `reportDocxPath`, `chartData` (base64 PNGs)
  - KEEP: `findingsJson`, `metadata`, `riskScore`, `complianceScores`, `narrativeSections`, `chatHistory`, `sanitizationMappings`, `denyListTerms`, `detectedLanguage`, `reportPdfJobId`, `reportPdfUrl`
- **PDF generation:** POST sanitized+de-sanitized HTML to Gotenberg `/forms/chromium/convert/html` instead of `/forms/libreoffice/convert`

### Python Architecture
- **KEEP:** `report_extraction_prompt.py` (findings extraction unchanged), `compliance_matrix.py` (static data), `report_theme.py` (add CSS color values)
- **MODIFY:** `report_narrative_prompt.py` (output format changes from plain text to HTML with CSS classes)
- **REWRITE:** `report_builder.py` (HTML assembly instead of python-docx DOCX construction)
- **REMOVE:** `chart_renderer.py` (matplotlib) — charts move to frontend Chart.js. Python still computes chart DATA (JSON configs) but doesn't render images
- **New endpoint:** Chart data computation endpoint returns Chart.js JSON configs instead of base64 PNGs

### Sanitization Architecture
- **HTML-first:** mammoth.js converts DOCX→HTML → Node walks HTML DOM → extracts text nodes → Presidio `/sanitize` per text segment → wraps in entity `<span>` tags
- **Counter map:** `Map<entityType, Map<originalValue, placeholderIndex>>` in Redis session. Ensures same original value always maps to same placeholder across re-sanitizations
- **Headers/footers:** docx_parser.py extracts these separately, sanitized alongside body text, stored in session as supplementary data for LLM extraction

## Boundaries

- No DOCX output — PDF only (clients only accept PDF)
- No matplotlib — all charts via Chart.js (HTML-native)
- No react-pdf — HTML preview via iframe replaces PDF preview
- No deny list UI — merged into mapping table
- No separate de-sanitization step — toggle in Review step
- Replace in-place — no v2 module, rewrite changed files, delete obsolete ones
- Same 5-step wizard structure as Phase 6
- LLM never sees de-sanitized values — de-sanitization is frontend-only
- Not user-customizable templates in v1 (same boundary as Phase 6)
- No report history/storage (same session-scoped model as Phase 6)

## Acceptance Criteria

1. Upload DOCX produces HTML preview with correct entity highlighting (incrementing placeholders per unique value)
2. Headers, footers, and text boxes are parsed and sanitized (not just body paragraphs)
3. Mapping table shows all detected entities with add/edit/delete capabilities
4. Text selection in HTML preview creates new entity mappings via popover with entity type dropdown
5. LLM generates styled HTML matching Template Executivo.pdf design (red branding, Roboto fonts, metric cards, section layouts)
6. Chart.js renders all chart types (pie, bar, stacked bar, radar, doughnut, horizontal bar) in both preview and PDF
7. Chat corrections update specific HTML sections in-place
8. De-sanitize toggle in Review step shows real values (frontend-only transform)
9. HTML→PDF via Gotenberg produces professional report with correct fonts, charts, and page breaks
10. No DOCX intermediary in the pipeline — HTML is the native format throughout
11. Generation progress display works (SSE stages visible to user)
12. Same auto-detect language, auto-trigger pipeline, session resume behaviors as Phase 6
