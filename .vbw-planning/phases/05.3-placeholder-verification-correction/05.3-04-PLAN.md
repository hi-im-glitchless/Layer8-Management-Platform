## Objective
Build the correction prompt infrastructure and regeneration pipeline. When a user selects text on the placeholder PDF and describes corrections in chat (e.g., "#1 should be {{title}}", "#2 remove this"), the LLM receives the current mapping plan + correction instructions and returns a full updated mapping as structured JSON. The backend then regenerates the DOCX with corrected placeholders and converts to PDF. This plan implements Decisions #4 (full pipeline regeneration), #5 (structured JSON correction response), #6 (full DOCX re-insertion + Gotenberg), #7 (clear selections after regeneration), and #8 (unlimited iterations).

## Context
- @sanitization-service/app/services/batch_mapping_prompt.py -- Existing batch mapping prompt builder. Correction prompt is a different module with different approach: receives full mapping plan as context, not just selections.
- @sanitization-service/app/services/analysis_prompt.py -- Reference template info loading pattern reused for correction prompt.
- @sanitization-service/app/services/insertion_prompt.py -- Pass 2 prompt builder. Regeneration calls this with the corrected mapping plan.
- @backend/src/services/templateAdapter.ts -- processChatFeedback() already has batch selection detection. Add correction mode alongside it. applyInstructions() is reused for regeneration.
- @backend/src/routes/templateAdapter.ts -- Chat SSE route. Add new event types for correction results.
- @frontend/src/features/adapter/types.ts -- ChatSSEEvent union. Add correction_result and regeneration_complete events.
- @frontend/src/features/adapter/components/StepVerify.tsx -- Created in Plan 03. Task 5 wires regeneration events into this component.
- Decision #4: Full pipeline regeneration on corrections (DOCX re-insertion + Gotenberg PDF)
- Decision #5: LLM returns structured JSON with full updated mapping (same schema as Phase 5)
- Decision #6: Full pipeline: update DOCX + Gotenberg re-convert to PDF (2-5 sec with spinner)
- Decision #7: Clear all selections after regeneration -- fresh PDF = fresh slate
- Decision #8: Unlimited iterations + explicit "Approve & Continue"

## Tasks

### Task 1: Correction Prompt Builder (Python)
**File(s):** `sanitization-service/app/services/correction_prompt.py`
**Action:** Create
**Details:**
- New module with:
  ```python
  def build_correction_system_prompt() -> str:
  ```
  - Role: "You are a template adaptation assistant correcting placeholder mappings in a document."
  - Instructions: receive current mapping plan + user corrections, return FULL updated mapping plan JSON
  - Output format: same as Pass 1 output -- `{ "entries": [...], "warnings": [...] }` with full mapping plan
  - Key rules: preserve all unchanged mappings, apply corrections as described, validate field names against GW field set
  ```python
  def build_correction_user_prompt(
      current_mapping_plan: dict,
      user_corrections: str,
      selections: list[dict] | None = None,
      doc_paragraphs: list[dict] | None = None,
  ) -> str:
  ```
  - Section 1: Current mapping plan (all entries with section_index, gw_field, marker_type, section_text)
  - Section 2: User corrections (verbatim text, may reference #N selections)
  - Section 3: Referenced selections context (if selections provided, show selected text + paragraph index)
  - Section 4: Document paragraphs around the selections (5 paragraphs before/after for context)
  - Section 5: Output format instructions (return complete updated mapping plan, not just changes)
- The prompt emphasizes: "Return the COMPLETE mapping plan with all entries. Modified entries should reflect corrections. Removed entries should be omitted. Added entries should be included."
**Commit:** `feat(adapter): add correction prompt builder for placeholder correction flow`

### Task 3: Backend Correction Chat Mode
**File(s):** `backend/src/services/templateAdapter.ts`
**Action:** Modify
**Details:**
- Add new function `processCorrectionChat()` (private async generator, similar to processBatchSelectionChat):
  1. Build correction prompt via Python `/adapter/build-correction-prompt`
  2. Call LLM with correction system + user prompts (streaming)
  3. Yield delta events for streaming text display
  4. After LLM completes: validate response via `/adapter/validate-mapping` (same as Pass 1 validation -- the correction returns a full mapping plan)
  5. If valid: update mapping plan in wizard state
  6. Yield `correction_result` SSE event with the updated mapping plan
  7. Trigger regeneration: call `applyInstructions()` with updated mapping plan to re-run Pass 2
  8. After DOCX regeneration: call `generatePlaceholderPreview()` (from Plan 01/03) to create new PDF
  9. Yield `regeneration_complete` SSE event with new pdfJobId
- Modify `processChatFeedback()`:
  - Add detection: if the wizard state currentStep is 'verify' AND message contains #N selections, use correction mode
  - The existing `isBatchMessage` check stays for backward compatibility but correction mode takes priority when in 'verify' step
  - Non-#N messages in verify step: still use standard chat flow (general questions about the mapping)
- Add helper `isInCorrectionMode(wizardState: WizardState): boolean`:
  - Returns true when currentStep === 'verify'
**Commit:** `feat(adapter): add correction chat mode with regeneration pipeline to processChatFeedback`

### Task 5: StepVerify Regeneration Wiring
**File(s):** `frontend/src/features/adapter/components/StepVerify.tsx`
**Action:** Modify
**Details:**
- Wire correction SSE events into StepVerify:
  1. Watch `chat.latestCorrectionResult`: update local mapping plan, call onMappingUpdate
  2. Watch `chat.regenerationResult`:
     - Update placeholderPdfJobId to trigger PDF polling for the new PDF
     - Update placeholderCount
     - Call selectionState.resetSelections() (Decision #7: clear selections on regeneration)
     - Show toast: "Placeholders updated -- review the changes"
  3. Show regeneration spinner: while correction is processing, show a small loading indicator overlay on the PDF area
     - "Regenerating..." text with Loader2 spinner

- Clear correction state when sending new chat message (chat.clearCorrectionState())
- The regeneration flow from user perspective:
  1. Select text on PDF, describe correction in chat
  2. LLM streams response, then correction_result arrives (mapping updated)
  3. Backend auto-regenerates DOCX + PDF (2-5 seconds)
  4. regeneration_complete arrives with new pdfJobId
  5. PDF viewer updates to show fresh PDF
  6. Previous selections are cleared
  7. User reviews and can make more corrections or approve
