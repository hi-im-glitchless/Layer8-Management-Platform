# Phase 5.3: Placeholder Verification & Correction — Discussion Context

## User Vision

Replace the manual select-and-describe mapping workflow (Phase 5.2) with an auto-map + verify/correct model. After DOCX upload, the LLM automatically runs analysis and placeholder insertion in one pass. The Verify step shows the PDF with all Jinja placeholders visible (styled with light backgrounds), and the user reviews the result. If corrections are needed, the user selects text on the PDF (same numbered selection pattern from 5.2), describes the fixes in chat, and the LLM returns an updated mapping as structured JSON. The backend regenerates the DOCX + PDF with corrected placeholders. The user iterates until satisfied, then clicks "Approve & Continue" to see the rendered preview with GW data.

This phase also restructures the wizard from 5 steps to 4: Upload → Verify Placeholders → Preview → Download. The old Analysis and Adaptation steps merge into the auto-map flow (processing shown in Upload step).

## Essential Features

- **LLM auto-mapping on upload**: After DOCX upload, LLM automatically runs Pass 1 (analysis) + Pass 2 (insertion) without user intervention. Progress shown in Upload step.
- **Placeholder verification PDF**: Analysis step replaced by Verify step showing PDF with all Jinja placeholders visible, styled with light background (light blue/grey) via DOCX paragraph shading before Gotenberg conversion.
- **Three correction types via chat**: (1) Select unmapped text → tell LLM it should be a placeholder, (2) Select wrong placeholder → tell LLM the correct placeholder, (3) Select unwanted placeholder → tell LLM to remove it. All expressed in natural language.
- **Reuse 5.2 selection infrastructure**: Same click-to-select numbered selection pattern (#1, #2, #3...) from Phase 5.2. LLM infers correction type from chat context — no mode buttons or structured format needed.
- **Structured JSON correction response**: LLM receives current mapping + correction message, returns full updated mapping as JSON (same schema as Phase 5). Backend applies complete mapping to regenerate DOCX.
- **Full pipeline regeneration**: Corrections trigger full DOCX re-insertion + Gotenberg PDF conversion (2-5 seconds with progress spinner). Always accurate — no in-place patching.
- **Clear selections after regeneration**: Fresh PDF = fresh slate. Old selections irrelevant after corrections applied. User makes new selections if more issues spotted.
- **Unlimited iterations**: No limits on correction/regeneration cycles. User clicks "Approve & Continue" when satisfied.
- **Placeholder navigation map**: Document structure browser repurposed from 5.2's paragraph list to a placeholder map. Lists all placeholders with document locations. Click to jump to position in PDF. Essential for long documents and hidden sections (headers, footers, empty cells).
- **Coverage counter**: Toolbar shows "X placeholders" count, updates after corrections. KB badge (brain +1) remains, fires on download.
- **4-step wizard**: Upload → Verify Placeholders → Preview → Download. Old 5-step flow simplified by merging Analysis + Adaptation into auto-map (runs in Upload step with progress indicators).
- **Translation as optional Step 4**: When Phase 6 is implemented, translation inserts between Preview and Download (4 → 5 steps). Auto-detect language (EN ↔ PT-PT), offer translation option.

## Technical Preferences

- Auto-map progress reuses existing progress infrastructure with all state persistence fixes (no reset on refresh/navigation)
- Placeholder styling implemented via DOCX paragraph/run shading (light background) before Gotenberg PDF conversion — same pipeline as Phase 5.1's annotated preview
- Chat panel always visible in Verify step, same PDF-left/chat-right layout as Phase 5.2
- LLM correction prompt includes current mapping plan as context; response is full updated mapping JSON
- Selection state from Phase 5.2 (pdfjs text layer, numbered selections, batch chat) reused directly for corrections
- Bad auto-maps shown as-is — the Verify step IS the quality gate. KB learning improves accuracy over time per template type

## Boundaries

- No green/yellow/orange shading from 5.1/5.2 — replaced by styled placeholder backgrounds (light blue/grey)
- No fallback to manual mapping — auto-map + verify is the only flow
- No quality gate before showing auto-map results — trust the correction loop
- No selection persistence between regenerations — clean slate each time
- No iteration limits or warnings — corrections are quick and targeted
- Phase 5.3 replaces Phase 5.2's workflow for the Analysis step (5.2 selection infra reused, but manual mapping flow retired)
- Phase 6 old planning artifacts are stale — Phase 6 will be re-discussed and re-planned from scratch to fit the new 4-step wizard

## Acceptance Criteria

1. DOCX upload triggers automatic LLM analysis + placeholder insertion with progress indicators
2. Verify step shows PDF with styled Jinja placeholders (light background, scannable)
3. User can select text/placeholders on PDF using 5.2's numbered selection pattern
4. User can describe corrections in chat using natural language ("#1 should be {{title}}", "#2 remove this")
5. LLM returns structured JSON with updated mapping; backend regenerates DOCX + PDF
6. Regeneration clears previous selections; user evaluates fresh PDF
7. User can iterate unlimited times (select → correct → regenerate → review)
8. "Approve & Continue" button advances to Preview step (triggers GW data render)
9. Placeholder navigation map lists all placeholders with click-to-jump
10. Coverage counter shows placeholder count in toolbar
11. Wizard is 4 steps: Upload → Verify → Preview → Download
12. Translation (Phase 6, future) inserts as optional Step 4 between Preview and Download
