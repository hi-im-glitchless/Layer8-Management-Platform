# Phase 05.1: Analysis Preview & Mapping Memory Discussion

**Date:** 2026-02-13
**Phase:** 5.1 (inserted after Phase 5)
**Goal:** Annotated document preview in Analysis step highlighting mapped vs missing placeholders, plus a persistent knowledge base that stores completed mappings as few-shot examples for future LLM analyses

## User Vision

After Phase 5 ships, the LLM maps sections well but misses some. The user wants a visual way to see what the LLM mapped (green), what it missed but the reference template expects (yellow auto-gaps), and the ability to manually mark additional sections the LLM missed entirely. Corrections feed into a persistent knowledge base so the LLM improves with each use. After 2-3 runs per template type, the gaps should largely disappear.

## Essential Features

### Annotated Preview (Step 2 Enhancement)
- Render annotated PDF in Step 2 using DOCX paragraph shading via python-docx + Gotenberg pipeline (reuses Phase 4)
- Green background shading on paragraphs with successful mappings
- Yellow background shading on auto-detected gaps (reference template fields missing from mapping plan)
- Read-only PDF with tooltips showing GW field name on mapped sections
- Gap detection by comparing mapping plan fields against reference template's known GW fields for the type+language combination

### Step 2 Layout Redesign
- Left panel: Tabbed view with "Preview" tab (annotated PDF) and "Mappings" tab (editable table)
- Right panel: Chat panel (unchanged from Phase 5)
- Default to Preview tab (visual-first experience)
- Tabs share the same data — different representations of the mapping state

### Inline Gap Correction
- Mapping table rows are directly editable for gap rows (click GW Field cell, type correct field)
- "Add Missing Section" button below the mapping table opens a searchable popover of unmapped paragraphs
- Unmapped paragraphs sourced from parsed DOCX structure (Phase 4 parser output) minus already-mapped entries
- User selects unmapped paragraph → new gap row appears in mapping table → user fills in GW field inline
- All corrections (inline edits + added sections) update the mapping plan in wizard state

### Mapping Memory (Knowledge Base)
- Persistent storage via Prisma model in database (SQLite dev / PostgreSQL prod)
- Shared across all users (2-5 person team, same template types)
- Section text stored as normalized patterns (lowercase, stripped whitespace, client-specific words removed)
- All confirmed mappings saved — no confidence threshold (user downloaded = user approved)
- Mappings persisted after successful download (Step 5 completion) as the strongest approval signal
- Deduplication by normalized section text + template type + language; update usage_count on repeats

### Few-Shot Prompt Injection
- 3-5 examples injected into future LLM analysis prompts
- Filtered by matching template_type + language
- Ranked by usage_count (most-confirmed patterns first)
- Injected as "## Previous Successful Mappings" section in the analysis prompt, after reference template patterns
- Silent learning — no badges or indicators showing which mappings came from the knowledge base

## Technical Preferences

### Preview Rendering Pipeline
- Apply paragraph background colors (green/yellow) in python-docx before Gotenberg conversion
- Use the existing Phase 4 pipeline: modified DOCX → Gotenberg → PDF → react-pdf (PdfPreview component)
- Paragraph index from mapping plan maps directly to DOCX paragraphs
- Tooltips via react-pdf annotation layer (lightweight overlay, no coordinate mapping for clicks)

### Gap Detection Algorithm
- Load reference template's known GW fields via reference_loader (Phase 5, 05-01)
- Compare mapping plan's gwField values against reference field set
- Missing fields = auto-detected gaps with yellow shading at expected paragraph positions
- If expected position unknown: append to gap list without positional highlight

### Knowledge Base Schema (Prisma)
- Model: `TemplateMapping`
- Fields: id, templateType, language, normalizedSectionText, gwField, markerType, confidence, usageCount, createdAt, updatedAt
- Unique constraint: (templateType, language, normalizedSectionText, gwField)
- Text normalization: lowercase, collapse whitespace, strip common client-specific terms

### Few-Shot Selection Query
- `WHERE templateType = ? AND language = ? ORDER BY usageCount DESC LIMIT 5`
- Format each as: `Section: "{normalizedSectionText}" → GW Field: {gwField} (confirmed {usageCount} times)`

### Wizard Integration
- Download endpoint triggers async knowledge base save (non-blocking)
- Read final mapping plan from wizard state (includes all corrections)
- Normalize section text, upsert into TemplateMapping table

## Boundaries

### In Scope (Phase 5.1)
- Annotated PDF preview in Step 2 with green/yellow paragraph shading
- Tabbed layout (Preview | Mappings) in Step 2
- Inline editing of gap rows in mapping table
- "Add Missing Section" paragraph picker
- TemplateMapping Prisma model and CRUD
- Few-shot injection into analysis prompt
- Knowledge base population on successful download

### Out of Scope (Phase 5.1)
- Admin UI for knowledge base management (add later if needed)
- Per-user knowledge bases (shared is sufficient for small team)
- Embedding-based similarity matching (normalized text is sufficient)
- Training mode or explicit "learning" UX (normal wizard flow)
- Confidence badges on knowledge-base-sourced mappings
- Clickable PDF highlights (paragraph picker handles missing section addition)

## Acceptance Criteria

1. Step 2 shows annotated PDF with green (mapped) and yellow (gap) paragraph shading
2. Step 2 has tabbed layout: Preview (default) and Mappings tabs
3. Gap rows in mapping table are inline-editable (GW field cell)
4. "Add Missing Section" button shows searchable list of unmapped paragraphs
5. User-added sections appear as new gap rows in the mapping table
6. Inline edits and added sections update the wizard state mapping plan
7. TemplateMapping Prisma model exists with unique constraint on (type, language, text, field)
8. Successful download triggers knowledge base persistence of all mappings
9. Subsequent analyses for same type+language include 3-5 few-shot examples from knowledge base
10. Gap detection compares mapping plan against reference template's known GW fields

## Decisions Made

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | DOCX shading + Gotenberg for preview highlights | Reuses Phase 4 pipeline, pixel-perfect, no frontend annotation complexity |
| 2 | Tabbed layout (Preview + Mappings), Chat stays right | Same data in two views, avoids panel compression |
| 3 | Default to Preview tab | Visual-first experience is the phase's primary deliverable |
| 4 | Gap detection via reference template field comparison | Precise, based on existing reference_loader data |
| 5 | Inline editing for gap rows in mapping table | Direct, fast correction — chat still available for complex changes |
| 6 | "Add Missing Section" button + searchable popover | Clean UX, keeps table as single correction interface |
| 7 | Prisma model for knowledge base (not Redis) | Persistent beyond session TTL, queryable, consistent with data layer |
| 8 | Shared knowledge base across all users | Small team (2-5), same template types, faster convergence |
| 9 | Normalized text patterns for matching | Handles case/whitespace variations across client templates |
| 10 | Persist after download (Step 5) | Strongest approval signal, captures all corrections |
| 11 | No confidence threshold for persistence | Download = approval regardless of original LLM confidence |
| 12 | 3-5 few-shot examples ranked by usage count | Balances context value vs token budget |
| 13 | Silent few-shot learning (no badges) | Simplest UX, system just gets better over time |
| 14 | Normal wizard flow (no training mode) | Knowledge base grows organically, no extra concepts to learn |
| 15 | No admin UI for now | Deferred until team finds need to prune entries |
| 16 | Read-only PDF with tooltips (no clickable highlights) | Paragraph picker handles missing section addition without PDF coordinate mapping |
