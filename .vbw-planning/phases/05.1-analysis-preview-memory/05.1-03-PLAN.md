## Objective
Wire the Python annotation service and knowledge base into the Node.js backend. This plan adds four capabilities: (1) an annotated preview endpoint that calls the Python /annotate service and pipes the result through Gotenberg for PDF conversion, (2) a KB persistence hook in the download endpoint that saves confirmed mappings after successful download, (3) a mapping update endpoint for inline edits from the frontend, and (4) the KB query integration in the analysis flow so few-shot examples are passed to the Python analyze endpoint. All backend templateAdapter.ts modifications are consolidated here to avoid same-wave file conflicts with Plan 4 (which only modifies Python files). Depends on Plan 1 (KB service) and Plan 2 (Python annotate endpoint).

**Commit:** `feat(adapter): add annotated preview service and wizard state extension`
**Files:**
- backend/src/services/wizardState.ts (modified) -- add WizardAnnotatedPreview interface and field
- backend/src/services/templateAdapter.ts (modified) -- add generateAnnotatedPreview() and KB query in analyzeTemplate()

**Details:**
1. Add to `wizardState.ts`:
   - `WizardAnnotatedPreview` interface: `{ pdfJobId: string | null; pdfUrl: string | null; tooltipData: Array<{ paragraphIndex: number; gwField: string; markerType: string; sectionText: string; status: 'mapped' | 'gap' }>; unmappedParagraphs: Array<{ paragraphIndex: number; text: string; headingLevel: number | null }>; gapSummary: { mappedFieldCount: number; expectedFieldCount: number; coveragePercent: number } | null }`
   - Add `annotatedPreview: WizardAnnotatedPreview` to WizardState interface with default null values
   - Update createWizardSession() default state with annotatedPreview defaults
   - Add annotatedPreview to the deep merge in updateWizardSession()
2. Add to `templateAdapter.ts`:
   - `generateAnnotatedPreview(wizardState: WizardState): Promise<WizardState>`:
     - Read mapping plan from wizardState.analysis.mappingPlan
     - Read template base64 from wizardState.templateFile.base64
     - POST to Python /adapter/annotate with {template_base64, mapping_plan (snake_case)}
     - Receive annotated_base64, tooltip_data, unmapped_paragraphs, gap_summary
     - Save annotated DOCX bytes to disk (uploads/documents/{uuid}_annotated.docx)
     - Queue PDF conversion via addPdfConversionJob()
     - Update wizard state with annotatedPreview data (pdfJobId, tooltipData, unmappedParagraphs, gapSummary)
3. Modify `analyzeTemplate()` in templateAdapter.ts:
   - Before POST to /adapter/analyze, call `queryFewShotExamples(templateType, language)` from templateMapping service
   - Convert results to snake_case format for Python: `{ normalized_section_text, gw_field, marker_type, usage_count }`
   - Add `few_shot_examples` field to the /adapter/analyze POST body
   - If KB query fails (DB error), log warning and continue with empty examples array (graceful degradation)

### Task 2: Annotated Preview & Mapping Update Routes
**Commit:** `feat(adapter): add annotated-preview and update-mapping endpoints`
**Files:**
- backend/src/routes/templateAdapter.ts (modified) -- add two new endpoints

**Details:**
1. Add `POST /api/adapter/annotated-preview`:
   - Requires auth, accepts { sessionId } body (Zod validated)
   - Get wizard state, verify currentStep is 'analysis' (annotated preview is generated during Step 2)
   - Call generateAnnotatedPreview(state)
   - Return 202 with { pdfJobId, tooltipData, unmappedParagraphs, gapSummary }
   - Uses same PDF polling pattern as existing preview endpoint (GET /api/adapter/preview/:sessionId already supports polling by jobId)
2. Add `GET /api/adapter/annotated-preview/:sessionId`:
   - Returns cached annotatedPreview data from wizard state (for page reload)
   - Includes current PDF status if pdfJobId exists
   - Includes tooltipData and unmappedParagraphs
3. Add `POST /api/adapter/update-mapping`:
   - Requires auth, body schema: `{ sessionId: uuid, updates: { editedEntries?: Array<{ sectionIndex: number, gwField: string, markerType: string }>, addedEntries?: Array<{ paragraphIndex: number, gwField: string, markerType: string }> } }`
   - Get wizard state, verify analysis.mappingPlan exists
   - For editedEntries: find matching entry by sectionIndex and update gwField + markerType, set confidence to 1.0 (user-confirmed)
   - For addedEntries: create new MappingEntry with confidence 1.0, lookup section text from paragraph index
   - Update mappingPlan in wizard state via updateWizardSession()
   - Return updated mapping plan
4. Zod validation for all input fields
