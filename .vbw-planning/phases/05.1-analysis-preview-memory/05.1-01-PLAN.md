## Objective
Create the persistent knowledge base data layer using Prisma. This stores confirmed template mappings so future LLM analyses can benefit from few-shot examples of previously successful mappings. The service handles text normalization, upsert-with-increment semantics, and few-shot query retrieval. This plan has no dependencies and runs in wave 1.

## Context
- @backend/prisma/schema.prisma -- current schema with User, TrustedDevice, AuditLog, DenyListTerm, LlmSettings models; add TemplateMapping here
- @backend/src/services/templateAdapter.ts -- existing orchestrator; will consume the KB service in Plan 3 (download hook) and Plan 4 (few-shot query)
- @frontend/src/features/adapter/types.ts -- MappingEntry and MappingPlan types define the shape of data that will be persisted
- @.vbw-planning/phases/05.1-analysis-preview-memory/05.1-CONTEXT.md -- decisions #6-11 specify Prisma model, shared scope, normalized text, upsert-on-download, 3-5 few-shot examples ranked by usageCount

## Tasks

### Task 1: Prisma TemplateMapping Model & Migration
**Commit:** `feat(adapter): add TemplateMapping Prisma model for knowledge base persistence`
**Files:**
- backend/prisma/schema.prisma (modified) -- add TemplateMapping model
- backend/prisma/migrations/YYYYMMDD_add_template_mapping/ (new) -- auto-generated migration

**Details:**
1. Add `TemplateMapping` model to `schema.prisma`:
   - `id` String @id @default(cuid())
   - `templateType` String (web|internal|mobile)
   - `language` String (en|pt-pt)
   - `normalizedSectionText` String -- lowercase, whitespace-collapsed section text
   - `gwField` String -- target GW field path (e.g. "client.short_name")
   - `markerType` String -- text|paragraph_rt|run_rt|table_row_loop|control_flow
   - `confidence` Float @default(1.0) -- last-seen confidence from mapping plan
   - `usageCount` Int @default(1) -- incremented on each duplicate upsert
   - `createdAt` DateTime @default(now())
   - `updatedAt` DateTime @updatedAt
   - `@@unique([templateType, language, normalizedSectionText, gwField])` -- composite unique constraint
   - `@@index([templateType, language])` -- query index for few-shot lookups
   - `@@index([usageCount])` -- sort index for ranking
2. Run `npx prisma migrate dev --name add_template_mapping` to generate and apply migration
3. Run `npx prisma generate` to update the Prisma client

### Task 2: Text Normalization Utility
**Commit:** `feat(adapter): add text normalization utility for knowledge base matching`
**Files:**
- backend/src/services/templateMapping.ts (new) -- KB service with normalization

**Details:**
1. Create `backend/src/services/templateMapping.ts`
2. Implement `normalizeSectionText(text: string): string`:
   - Trim leading/trailing whitespace
   - Convert to lowercase
   - Collapse consecutive whitespace (spaces, tabs, newlines) into single space
   - Strip common client-specific filler patterns: placeholder underscores ("___"), ellipsis ("..."), em-dash patterns
   - Return the normalized string
3. Export the function for use by the upsert and query operations
4. Add Zod schemas for service inputs:
   - `templateMappingSchema` for single mapping input
   - `fewShotQuerySchema` for query parameters

### Task 3: Upsert & Bulk Upsert Operations
**Commit:** `feat(adapter): implement upsert and bulk operations for template mapping KB`
**Files:**
- backend/src/services/templateMapping.ts (modified) -- add upsert operations

**Details:**
1. Implement `upsertMapping(input: { templateType, language, sectionText, gwField, markerType, confidence })`:
   - Normalize sectionText via normalizeSectionText()
   - Use Prisma `upsert` with the composite unique key (templateType, language, normalizedSectionText, gwField)
   - On create: set usageCount=1, confidence=input.confidence
   - On update: increment usageCount by 1, update confidence to latest value, touch updatedAt
   - Return the upserted record
2. Implement `bulkUpsertMappings(mappings: Array<{...}>)`:
   - Accepts array of mapping entries (from a completed wizard download)
   - Runs all upserts in a Prisma transaction for atomicity
   - Returns count of created and updated records
   - Non-blocking: caller can fire-and-forget (errors logged, not thrown to user)
3. Add error handling: log failures but do not throw (KB persistence is non-blocking, CONTEXT.md decision #9)

### Task 4: Few-Shot Query Service
**Commit:** `feat(adapter): implement few-shot query service for template mapping KB`
**Files:**
- backend/src/services/templateMapping.ts (modified) -- add query function

**Details:**
1. Implement `queryFewShotExamples(templateType: string, language: string, limit?: number)`:
   - Default limit = 5 (CONTEXT.md decision #12: 3-5 examples)
   - Query: `WHERE templateType = ? AND language = ? ORDER BY usageCount DESC LIMIT ?`
   - Return array of `{ normalizedSectionText, gwField, markerType, usageCount }`
   - Format each as a string for prompt injection: `Section: "{normalizedSectionText}" -> GW Field: {gwField} (confirmed {usageCount} times)`
2. Export `formatFewShotExamples(examples: TemplateMappingRecord[]): string`:
   - Formats the array into a prompt section: "## Previous Successful Mappings\n\nThese mappings were confirmed by users in previous template adaptations:\n\n" followed by numbered list
   - Returns empty string if no examples found (no section injected)
