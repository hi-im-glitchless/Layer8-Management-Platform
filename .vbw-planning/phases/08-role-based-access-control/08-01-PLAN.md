## Objective

Replace the binary `isAdmin: Boolean` column with a 4-tier `Role` enum (`NORMAL`, `PM`, `MANAGER`, `ADMIN`) in the Prisma schema. Generate and customize the migration to preserve existing data (isAdmin=true becomes ADMIN, isAdmin=false becomes NORMAL). Update the seed script.

## Context

- `@backend/prisma/schema.prisma` -- User model has `isAdmin Boolean @default(false)` at line 16; needs Role enum and field replacement
- `@backend/src/scripts/seed-admin.ts` -- creates admin user with `isAdmin: true` at line 34; needs `role: 'ADMIN'`

## Tasks

### Task 1: Add Role enum and replace isAdmin in schema.prisma

**Scope:** `backend/prisma/schema.prisma`

**Action:**

1. Add a new enum above the User model:
   ```prisma
   enum Role {
     NORMAL
     PM
     MANAGER
     ADMIN
   }
   ```

2. In the `User` model, replace:
   ```prisma
   isAdmin Boolean @default(false)
   ```
   with:
   ```prisma
   role Role @default(NORMAL)
   ```

3. No other models reference `isAdmin`, so no further schema changes needed.

**Acceptance:** `npx prisma validate` passes. Schema shows Role enum and User.role field. No isAdmin field exists.

### Task 2: Generate Prisma migration with data migration SQL

**Scope:** `backend/prisma/migrations/`

**Action:**

1. Run `npx prisma migrate dev --name rbac-role-system --create-only` to generate the migration SQL without applying it.

2. SQLite doesn't support ALTER TABLE ADD COLUMN with enum — Prisma will handle this by recreating the table. Open the generated migration SQL file and ensure the data migration is correct. Prisma's SQLite migration for dropping a column and adding a new one typically:
   - Creates a new table with the updated schema
   - Copies data from the old table
   - Drops the old table
   - Renames the new table

3. In the generated SQL, find the INSERT INTO statement that copies data and ensure the role column is populated correctly. If Prisma's default migration doesn't handle the data mapping, add a custom SQL step **before** the table drop:
   ```sql
   -- After the new table is created and data copied with default NORMAL:
   UPDATE "new_User" SET "role" = 'ADMIN' WHERE "id" IN (SELECT "id" FROM "User" WHERE "isAdmin" = 1);
   ```
   The exact SQL depends on what Prisma generates. The key requirement: users with `isAdmin=true` get `role='ADMIN'`, all others get `role='NORMAL'`.

4. Apply the migration: `npx prisma migrate dev`

5. Regenerate the Prisma client: `npx prisma generate`

**Acceptance:** Migration applied successfully. `SELECT id, role FROM User` returns ADMIN for previous admin users and NORMAL for others. No isAdmin column exists.

### Task 3: Update seed script

**Scope:** `backend/src/scripts/seed-admin.ts`

**Action:**

1. Replace `isAdmin: true` with `role: 'ADMIN'` in the `prisma.user.create` call (line 34).
2. Remove any other references to `isAdmin` in this file.

**Acceptance:** `npx tsx backend/src/scripts/seed-admin.ts` creates an admin user with `role: 'ADMIN'` (or reports user already exists).

## Verification

```bash
# Schema validates
cd backend && npx prisma validate

# Migration applied
cd backend && npx prisma migrate status

# Seed works
cd backend && npx tsx src/scripts/seed-admin.ts

# Role enum exists in generated client
grep -r "enum Role" backend/node_modules/.prisma/client/index.d.ts
```

## Success Criteria

- Role enum with 4 values exists in schema
- isAdmin column completely removed from User model
- Existing admin users have role=ADMIN after migration
- Existing non-admin users have role=NORMAL after migration
- Seed script uses role field
