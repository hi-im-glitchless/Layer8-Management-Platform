# Phase 5.2: Interactive PDF Mapping — Discussion Context

## User Vision

Replace the table-based mapping UI (from Phase 5) with a PDF-first, select-and-describe workflow. Users interact directly with a continuously rendered PDF, selecting text passages and numbering them (#1, #2, #3...), then describe all selections in a single free-form chat message. The LLM resolves each selection to a Ghostwriter field and marker type. Only confirmed mappings appear as green highlights — no yellow gap shading. A document structure browser handles blank/invisible paragraphs that can't be highlighted on PDF. The knowledge base learns per template type (internal/web/mobile), improving auto-mapping accuracy over time.

## Essential Features

- **PDF text selection**: Users highlight text on the continuously rendered PDF; each selection is auto-added as the next numbered item (#1, #2, #3...) with a toast confirmation
- **Document structure browser**: Collapsible sidebar toggled from the PDF toolbar, listing all DOCX paragraphs including blank/invisible ones (headers, footers, empty table cells) for selection
- **Batch chat description**: Free-form natural language message referencing selection numbers (e.g., "#1 is the finding title, #2 is severity, #3 is the executive summary footer")
- **LLM batch resolution**: LLM parses numbered references, infers GW field + marker type for each selection in a single response
- **Inline confirmation**: Each resolved mapping appears as an overlay/tooltip on the PDF at the selection location showing GW field + marker type; user clicks checkmark to confirm (turns green) or X to reject (turns orange); batch confirm-all button available
- **Rejection recovery**: Rejected selections stay numbered and turn orange; user re-describes them in chat (e.g., "#2 is actually the remediation steps"). Multiple corrections can be batched in one message. LLM re-maps only the specified selections
- **Three visual states**: Blue outline (pending/freshly selected), green fill (confirmed), orange outline (rejected/needs correction)
- **Coverage counter**: Minimal "X mapped" counter in toolbar showing progress without prescribing gaps
- **KB feedback badge**: Subtle brain icon with +1 animation when a mapping is saved to the knowledge base
- **Green-only highlighting**: Only confirmed mappings get green shading on the PDF; no yellow/gap highlights
- **KB per template type**: Knowledge base stores mappings filtered by templateType (internal/web/mobile) for few-shot reuse

## Technical Preferences

- PDF text layer from pdfjs-dist provides selectable `<span>` elements over canvas for text selection
- Selections tracked as numbered list in component state with text content + page/position metadata
- Free-form chat parsed by LLM — no structured input format required from user
- Inline overlays positioned using PDF text layer coordinates from selection metadata
- Structure browser reads DOCX paragraph data already parsed in session (from Phase 4/5 pipeline)

## Boundaries

- No yellow gap shading — coverage counter is informational only, not prescriptive
- No training mode — KB grows organically from normal wizard completions
- No admin UI for KB management (deferred)
- No auto-mapping on upload — user drives all selections manually (KB improves LLM accuracy, not automation)
- This phase modifies the Analysis step (Step 2) of the existing wizard; other steps unchanged

## Acceptance Criteria

1. User can select text passages on the PDF and see them auto-numbered (#1, #2, #3...)
2. User can open structure browser sidebar and select blank/invisible paragraphs
3. User can describe all selections in one free-form chat message
4. LLM resolves all selections to GW fields in a single response
5. Each mapping appears inline on the PDF with accept/reject controls
6. Confirmed mappings turn green on the PDF; rejected ones turn orange and retain their selection number
7. User can re-describe rejected mappings in chat and LLM re-maps only those selections
8. Multiple corrections can be batched in one chat message (same pattern as initial description)
9. Coverage counter shows "X mapped" in toolbar
10. KB badge animates when mappings are persisted
11. KB stores mappings per template type and injects as few-shot examples in future analyses
