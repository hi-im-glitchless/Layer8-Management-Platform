## Relevant Patterns

1. **Normalization as canonical form** — deterministic, handles formatting variations between DOCX uploads
2. **Zone-aware querying** — queryByZone() already groups by zone; blueprints are zone-specific
3. **Fire-and-forget persistence** — non-blocking KB writes after download; errors logged not thrown
4. **zoneMap dependency** — regenerateWithLLM() populates wizardState.adaptation.zoneMap from Python response; used by persistMappingsToKB()
5. **Boilerplate filtering** — StyleHint model filters decorative styles from doc structure; keep for clean prompts

## Risks

1. **Schema migration:** 126 entries need zone backfill before unique constraint change. Use Prisma transaction, backfill from blueprints, log all changes
2. **Delete-and-recreate atomicity:** If creation fails after deletion, entry lost. Use Prisma transaction for atomic delete+create
3. **Inline annotation format:** Python doc structure builder needs new input (locked/unknown arrays). Backend must split before calling Python
4. **zoneMap gaps:** If sectionIndex not in zoneMap, defaults to "unknown". Log warnings, monitor distribution
5. **Boilerplate style filtering interaction:** Locked sections with boilerplate styles should still appear (they're confirmed mappings). Don't filter locked sections

## Recommendations

1. **Implementation order:** Schema migration → Backend lookup logic → Python annotated prompt → Remove cross-type/budget → Persistence changes → Frontend source field + badge
2. **Backfill strategy:** Query BlueprintPattern to infer zones; use field-type heuristics for remainder; delete duplicates on conflict
3. **Test with actual data:** Run migration on copy of dev.db before production; verify no constraint violations
4. **Keep boilerplate filtering** for unknown sections only; locked sections bypass filtering (they're confirmed correct)
5. **Prisma transactions** for all delete-and-recreate operations in correction flow
