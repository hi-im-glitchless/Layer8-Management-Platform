# Phase 5.6: Prescriptive Knowledge Base — Context

## User Vision

Transform the KB from an advisory few-shot prompt enhancer into a deterministic mapping cache with LLM fallback. When a document is uploaded, each section is looked up in the KB first — if a high-confidence match exists, that mapping is locked in without LLM involvement. Only genuinely unknown sections go to the LLM. Re-uploading a previously mapped document should produce near-identical correct results with zero LLM calls for known sections.

## Essential Features

1. **Prescriptive KB lookup** — Before LLM analysis, normalize each section's text and look up `(normalizedText + zone + templateType + language)` in the KB. Entries with confidence >= 0.8 are locked.
2. **Exact match only** — No fuzzy or embedding-based matching. Normalization (lowercase, strip whitespace/placeholders) handles minor variations. Templates are structurally stable across reuse.
3. **Inline annotations in prompt** — Locked and unknown sections presented in document order: `[RESOLVED: finding.title]` and `[NEEDS MAPPING]`. LLM sees positional context so it can infer unknowns from surrounding resolved sections.
4. **Delete-and-recreate corrections** — When a user overrides a locked mapping, delete the wrong entry entirely and create the correct one at 1.0. No decay, no ghost entries, immediate lock on re-upload.
5. **Summary banner** — Show "X/Y sections resolved from KB, Z analysed by LLM" above the mapping table.
6. **KB badge on rows** — Add `source: 'kb' | 'llm'` field to mapping entries, purely for CSS conditional styling (muted row + "KB" badge). No functional changes to mapping table data flow, events, or PDF pipeline.
7. **Full reconciliation on download** — Diff final mapping plan against what was locked. Unchanged locked entries get usageCount++. Changed locked entries get delete+recreate. New LLM entries created at 1.0.
8. **Dead entry pruning** — One-time migration to remove entries below confidence 0.3 (with log). Ongoing auto-prune during each persistMappingsToKB() cycle.

## Technical Preferences

- **Match key**: `(normalizedSectionText + zone + templateType + language)` — zone is part of the lookup to prevent cross-zone collisions
- **Lock threshold**: confidence >= 0.8
- **New entry confidence**: 1.0 (locks immediately on re-upload)
- **Lookup layer**: Backend does KB lookup via Prisma, splits sections into locked/unknown arrays, passes both to Python. Python builds the annotated prompt.
- **Repeating sections**: Match by text+zone, apply to all occurrences (findings loop pattern)
- **Cold start**: Full LLM analysis. No cross-type fallback — remove that code entirely. One LLM pass per new template type, ever after that it's cached.
- **Prompt structure**: Inline `[RESOLVED: field]` / `[NEEDS MAPPING]` annotations in document structure listing. Keep blueprints section for structural patterns. Remove: zone distribution, repetition hints, confidence notes, token budget enforcer.

## Schema Changes

1. **Add zone to unique constraint**: `@@unique([templateType, language, normalizedSectionText, gwField, zone])` — Prisma migration required
2. **Backfill NULL zones**: Before migration, backfill existing 126 entries' zone values from blueprint patterns. Ambiguous entries default to most likely zone or get deleted (recreated on next upload). Log all changes.
3. **Add source field**: `source: 'kb' | 'llm'` on mapping entry type (TypeScript only, not DB column — render-only field)

## Boundaries

- **No fuzzy matching** — Exact normalized match only. No token-overlap, no embeddings, no sentence-transformers.
- **No template fingerprinting** — Deferred to a future phase. Per-section lookup is sufficient for now.
- **No functional UI changes** — KB badge and muted styling are CSS-only. The mapping table data model, event handlers, PDF highlight/delete/regenerate pipeline remain untouched. If implementing the badge requires changing any of those, stop and flag.
- **No cross-type fallback** — Remove the existing cross-type fallback code. Each template type stands alone.
- **No new dependencies** — This phase uses existing SQLite/Prisma/Python stack only.

## Acceptance Criteria

1. Re-uploading a previously correctly mapped document produces the same correct mapping with zero LLM calls for known sections
2. Summary banner shows accurate KB vs LLM split counts
3. KB badge appears on locked rows with muted styling, no functional behavior changes
4. Overriding a locked mapping deletes the old entry and creates the correct one at 1.0 (immediate re-lock on next upload)
5. Dead entries (confidence < 0.3) are pruned in migration and auto-pruned ongoing
6. Zone is part of the unique constraint and match key
7. Cross-type fallback code is removed
8. Old KB context block (zone distribution, repetition hints, confidence notes) and token budget enforcer are removed from analysis_prompt.py
9. Blueprints section retained in prompt for structural pattern context
10. Only genuinely unknown sections are sent to the LLM, with locked sections as inline positional annotations

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Zone in match key | 'Description' in header vs body maps to different fields |
| Exact match only | 3-5 templates reused dozens of times; normalization handles variation |
| Inline annotations over separate KB block | Document order matters; LLM infers unknowns from surrounding resolved sections |
| Delete-and-recreate on correction | User explicitly says "this is wrong, this is right" — no ambiguity, no decay |
| Lock at 0.8, start at 1.0 | First correct mapping locks immediately; pragmatic for reuse-heavy scenario |
| Backend splits, Python builds | Clean separation: data logic in Node/Prisma, prompt logic in Python |
| Remove cross-type fallback | Dead weight with prescriptive model; one LLM pass per new template type suffices |
| Keep blueprints, remove rest | Blueprints encode structural knowledge annotations don't carry; rest is noise now |
| Remove token budget enforcer | No separate KB block to budget; annotations are part of document structure |
| Full reconciliation on download | Closes the feedback loop: diff final plan vs locked, apply delete+recreate consistently |
| Source field is render-only | No functional changes to mapping table pipeline; CSS badge only |
