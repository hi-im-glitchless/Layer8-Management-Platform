## Objective
Add zone to the TemplateMapping unique constraint so that the same normalized section text in different zones (e.g., header vs body) maps to different GW fields without collision. Before migrating, backfill existing entries with zone='unknown' to real zone values inferred from BlueprintPattern data, and prune dead entries (confidence < 0.3) to reduce noise. Update all Prisma where clauses that use the old 4-field compound key to use the new 5-field key. This is the foundational data model change that all subsequent plans depend on.

## Context
- @backend/prisma/schema.prisma -- Lines 91-110: TemplateMapping model. Current unique constraint at line 106: `@@unique([templateType, language, normalizedSectionText, gwField])`. Zone field exists at line 100 with `@default("unknown")`. Must change unique to include zone and update where clause names throughout the codebase.
- @backend/src/services/templateMapping.ts -- Lines 134-141: upsertMapping() uses `templateType_language_normalizedSectionText_gwField` as the where clause compound key. Lines 217-224: bulkUpsertMappings() uses the same key. Both must be updated to the new 5-field compound key `templateType_language_normalizedSectionText_gwField_zone`. Lines 327-353: queryByZone() and other query functions remain unchanged (they query by templateType+language, not by unique key).
- @backend/src/services/templateAdapter.ts -- Lines 969-1073: persistMappingsToKB() calls upsertMapping() which uses the unique constraint. The zone is already being passed from the zoneMap (line 1002-1013), so the data is correct; only the unique key lookup changes.
- @backend/prisma/migrations/ -- One existing migration (20260214143704_add_kb_intelligence). The new migration adds zone to the unique constraint.
- Research finding: 126 entries need zone backfill. BlueprintPattern stores zone+gwField associations that can be used to infer zones for existing entries.

## Tasks

### Task 1: Prune Dead Entries & Backfill Zone Values
**File(s):** `backend/prisma/backfill-zones.ts` (NEW)
**Action:** Create
**Details:**
- Create a standalone TypeScript script that runs with `npx tsx backend/prisma/backfill-zones.ts`
- Import prisma client from `@/db/prisma.js` (or create a standalone client import)
- Step 1 -- Prune dead entries:
  ```typescript
  const pruned = await prisma.templateMapping.deleteMany({
    where: { confidence: { lt: 0.3 } },
  });
  console.log(`[backfill] Pruned ${pruned.count} dead entries (confidence < 0.3)`);
  ```
- Step 2 -- Backfill zones:
  - Fetch all TemplateMapping entries where `zone === 'unknown'`
  - Fetch all BlueprintPattern entries grouped by templateType
  - For each unknown-zone entry:
    1. Look up BlueprintPattern entries matching `templateType` that contain the entry's `gwField` in their markers JSON
    2. If found, use the blueprint's zone value
    3. If not found, apply field-based heuristics: fields like `title`, `client.short_name`, `report_date`, `team[0].name` commonly appear in header/footer/cover -- map these to 'header'; finding-related fields map to 'body'
    4. If still ambiguous, default to 'body'
  - Log each change: `[backfill] Entry ${id}: zone unknown -> ${newZone} (gwField: ${gwField}, method: ${method})`
- Step 3 -- Handle potential duplicates after backfill:
  - After zone update, check for entries that would violate the new 5-field unique constraint (same templateType+language+normalizedSectionText+gwField+zone)
  - If duplicates exist, keep the one with highest usageCount and delete the rest
  - Log: `[backfill] Removed ${count} duplicate entries after zone assignment`
- Disconnect prisma at end
**Commit:** `chore(adapter): add zone backfill and dead entry prune script`

  ```prisma

  ```
- Keep all existing indexes unchanged (lines 107-109)

  2. Create a new unique index on (templateType, language, normalizedSectionText, gwField, zone)
- Important: The backfill script (Task 1) MUST run before this migration to avoid constraint violations from entries with zone='unknown' that would become duplicates
