## Objective
Build the Python-side foundation for LLM-powered placeholder placement. This plan creates the zone-aware prompt builder that produces structured DOCX representations for the LLM, and the validation endpoint that checks LLM-generated placement instructions against the actual document before application. The existing InstructionApplier is reused as-is for applying validated instructions. This addresses the "structured zone map input" and "placement instructions output" decisions from the phase context.

## Context
- @sanitization-service/app/services/insertion_prompt.py -- 127 lines. Existing Pass 2 prompt builder with build_insertion_prompt() and build_insertion_system_prompt(). The new placement prompt builder follows the same pattern but adds zone context and stricter rules. We create a NEW file (placement_prompt.py) rather than modifying this one, because the old insertion prompt is still used by autoMapTemplate() until Plan 05.5-03 unifies the pipelines.
- @sanitization-service/app/services/docx_parser.py -- 360 lines. DocxParserService.parse() returns DocxStructure with zone-tagged paragraphs (cover, body, header, footer, table_cell). Each paragraph has: text, style_name, heading_level, alignment, runs, zone, table_index. Sections include header/footer paragraphs. This is the input for the zone map builder.
- @sanitization-service/app/models/docx.py -- DocxStructure, DocxParagraph (with zone field), DocxTable, DocxSection models. ZoneType = "header" | "footer" | "body" | "table_cell" | "text_box" | "cover" | "unknown".
- @sanitization-service/app/models/adapter.py -- MappingEntry, MappingPlan, Instruction, InstructionSet, ApplyResponse models. FIELD_MARKER_MAP dict for marker type validation.
- @sanitization-service/app/services/instruction_applier.py -- InstructionApplier with 4 actions: replace_text, insert_before, insert_after, wrap_table_row. Sorts by paragraph_index descending. The validate-placement endpoint produces InstructionSet that feeds directly into this applier.
- @sanitization-service/app/routes/adapter.py -- 1327 lines. POST /apply endpoint validates + enriches + applies InstructionSet. The new validate-placement endpoint produces an InstructionSet compatible with this existing /apply endpoint.
- Decision #2: Structured zone map + paragraph text as LLM input
- Decision #3: Placement instructions (JSON) as LLM output -- validate before touching DOCX
- Decision #4: Skip and warn on mismatches

### Task 3: Placement Prompt Builder (System + User Prompts)
**File(s):** `sanitization-service/app/services/placement_prompt.py`
**Action:** Modify
**Details:**
- Implement `build_placement_system_prompt() -> str`:
  - Define role: "You are a DOCX template engineer specializing in Jinja2 placeholder placement"
  - Critical rules:
    1. Each instruction's original_text MUST be an exact substring of the paragraph text at paragraph_index
    2. Only replace the specific content that maps to a GW field -- preserve surrounding text
    3. Use the correct marker_type from the mapping plan (text, paragraph_rt, run_rt, table_row_loop, control_flow)
    4. For paragraph_rt fields (descriptions, impacts), replace the entire paragraph content
    5. For table_row_loop fields, use wrap_table_row action
    6. When multiple fields map to the same paragraph, generate separate instructions for each
    7. Include a confidence score (0-1) for each instruction
    8. Return ONLY valid JSON, no markdown fences or explanation
- Implement `build_placement_prompt(doc_structure: DocxStructure, mapping_plan: MappingPlan) -> str`:
  - Section 1: Zone map (from build_zone_map)
  - Section 2: Mapping entries to place (from mapping_plan.entries with gw_field, placeholder_template, marker_type, section_text hint)
  - Section 3: Required output format -- InstructionSet JSON with instructions array, each having action, paragraph_index, original_text, replacement_text, marker_type, gw_field, confidence
  - Section 4: Placement rules -- how to handle each marker type, zone-specific guidance (e.g., headers/footers often have run_rt fields)
**Commit:** `feat(adapter): implement placement prompt builder with zone context and placement rules`

### Task 4: Validate Placement Endpoint
**File(s):** `sanitization-service/app/routes/adapter.py`
**Action:** Modify
**Details:**
- Import new models: PlacementPromptRequest, PlacementPromptResponse, ValidatePlacementRequest, ValidatePlacementResponse
- Add `POST /build-placement-prompt` endpoint:
  - Decode base64, validate DOCX magic bytes
  - Parse DOCX via _parser.parse()
  - Build prompts via build_placement_system_prompt() and build_placement_prompt()
  - Return PlacementPromptResponse with prompt, system_prompt, paragraph_count
- Add `POST /validate-placement` endpoint:
  - Decode base64, parse DOCX to get actual paragraph texts
  - Parse LLM JSON response (strip markdown fences)
  - For each instruction:
    1. Validate paragraph_index is within bounds (0 to paragraph_count-1)
    2. Validate action is one of: replace_text, insert_before, insert_after, wrap_table_row
    3. Validate marker_type is valid
    4. Validate gw_field exists in FIELD_MARKER_MAP (warn if unknown)
    5. For replace_text: verify original_text is a substring of actual paragraph text at paragraph_index; if not found, try text-based relocation across all paragraphs
    6. For replace_text with empty original_text on paragraph_rt: use full paragraph text as original
    7. Filter instructions with confidence < 0.5 to warnings
  - Build valid InstructionSet from passing instructions
  - Return ValidatePlacementResponse with valid flag, instruction_set, applied/skipped counts, warnings
  - If zero instructions pass validation, return valid=false with errors
**Commit:** `feat(adapter): add build-placement-prompt and validate-placement endpoints`
