## Objective
Unify the auto-map and regeneration pipelines so both use the same LLM placement logic (regenerateWithLLM). Enrich KB persistence with zone context derived from LLM placement decisions. After this plan, there is a single code path for all placeholder placement, and the KB receives structural context (zone, marker type, co-occurrence patterns) instead of zone='unknown'. This addresses decisions #6 (unify auto-map and regeneration) and #7 (enrich KB from LLM placement decisions).

## Context
- @backend/src/services/templateAdapter.ts -- Lines 647-682: autoMapTemplate() currently chains analyzeTemplate() (Pass 1) + applyInstructions() (Pass 2). Must change Pass 2 to use regenerateWithLLM(). Lines 978-1079: persistMappingsToKB() hardcodes zone='unknown' at line 1008 and 1019. Must derive zone from doc structure or instruction set. Lines 1088-1167: persistBlueprintsAndHints() calls Python /detect-blueprints -- already works but gets 'unknown' zone data.
- @backend/src/services/wizardState.ts -- WizardState.adaptation.instructions stores the instruction set. regenerateWithLLM() from Plan 05.5-02 stores the validated instruction set here. persistMappingsToKB() can read it to map paragraph_index -> zone.
- @sanitization-service/app/routes/adapter.py -- POST /adapter/document-structure returns paragraphs with zone info. POST /adapter/build-placement-prompt also parses the DOCX. We can get zone data from the doc structure cached in wizard state.
- Decision #6: Unify auto-map and regeneration pipelines -- same LLM placement logic for both
- Decision #7: Enrich KB from LLM placement decisions -- zone + marker type context
