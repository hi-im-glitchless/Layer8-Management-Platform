## Objective
Build the TypeScript backend function that orchestrates the LLM placement pipeline for regeneration. This replaces the `reapplyFromMappingPlan()` call in the `/reapply` route with a new `regenerateWithLLM()` function that sends the DOCX + mapping plan to the LLM for intelligent placement. The function calls the Python endpoints built in Plan 05.5-01 (build-placement-prompt, validate-placement, apply). This plan can be built in parallel with Plan 05.5-01 since it only needs to know the API contract (request/response shapes), not the implementation.

The new regenerateWithLLM() replaces this with: build-placement-prompt -> LLM -> validate-placement -> apply. Lines 448-566: applyInstructions() shows the pattern for calling LLM + Python apply (same 5-step flow used here). Lines 419-434: mappingPlanToSnakeCase() helper for Python service calls. - Decision #5: Replace mechanical approach entirely -- no fallback

### Task 2: New regenerateWithLLM() Function
**File(s):** `backend/src/services/templateAdapter.ts`
**Action:** Modify
**Details:**
- Add new interface for Python validate-placement response:
  ```typescript
  interface ValidatePlacementResponse {
    valid: boolean;
    instruction_set: Record<string, unknown> | null;
    applied_count: number;
    skipped_count: number;
    warnings: string[];
    errors: string[];
  }
  ```
- Add new interface for Python build-placement-prompt response:
  ```typescript
  interface PlacementPromptResponse {
    prompt: string;
    system_prompt: string;
    paragraph_count: number;
  }
  ```
- Implement `regenerateWithLLM(wizardState: WizardState): Promise<WizardState>`:
  1. Extract mappingPlan, templateBase64 from wizard state
  2. Call Python POST /adapter/build-placement-prompt with template_base64 + mapping_plan (snake_case)
  3. Call LLM with system_prompt + prompt (non-streaming, collect full JSON response via generateStream)
  4. Call Python POST /adapter/validate-placement with llm_response + template_base64 + template_type + language + paragraph_count
  5. If validation returns valid=false or no instruction_set, throw error with details from errors array
  6. Call existing Python POST /adapter/apply with template_base64 + validated instruction_set
  7. Save adapted DOCX to disk (same pattern as applyInstructions())
  8. Update wizard state with appliedCount, skippedCount, placementWarnings (from validation warnings + apply warnings)
  9. Return updated wizard state
- Error handling: on LLM failure, log error and throw with descriptive message. Do NOT modify wizard state on failure (preserves last good state for retry). On validation failure with zero valid instructions, throw error listing the validation errors.
- Use maxTokens: 8192 for the LLM call (same as applyInstructions Pass 2)
- Use feature: 'template-adapter' for model routing
**Commit:** `feat(adapter): implement regenerateWithLLM() LLM placement pipeline`
