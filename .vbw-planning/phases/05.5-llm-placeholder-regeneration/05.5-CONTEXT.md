# Phase 5.5: LLM-Powered Placeholder Regeneration — Context

## User Vision

Replace the brittle mechanical find-and-replace regeneration engine with an LLM-based approach. The current `apply-from-mapping` endpoint does text matching and paragraph-index lookups that break documents when PDF indices don't align with DOCX indices. The LLM reads the actual document, understands its structure, and places placeholders intelligently.

The mapping table UI stays **exactly as-is**. Users still add, edit, and remove mappings in the same table. Only the backend regeneration path changes.

## Essential Features

1. **Full regeneration scope** — Every regeneration processes the ENTIRE mapping table, not just changed entries. The LLM sees the full picture for better placement decisions.
2. **Unified pipeline** — Both initial auto-map (on upload) and user-triggered regeneration use the same LLM placement pipeline. One code path to maintain.
3. **Structured zone map input** — The LLM receives a structured representation of the DOCX: zones (header, footer, body sections, tables) with numbered paragraphs and their text content. Uses the KB's zone-aware parser output.
4. **Placement instructions output** — The LLM returns structured JSON instructions: which paragraph, what original text to find, what placeholder to insert. The sanitization service validates and applies them.
5. **KB enrichment from placements** — Each LLM placement decision feeds structural context back to the KB (zone, marker type, co-occurrence patterns). Future auto-maps get better structural hints.
6. **Skip-and-warn error handling** — If an LLM instruction doesn't match actual document content (e.g., paragraph text doesn't contain the expected string), skip it, apply the rest, and show warnings to the user.

## Technical Preferences

- No mechanical fallback — the old `apply-from-mapping` text-replacement logic is fully replaced
- LLM placement pipeline serves both auto-map and regeneration
- Zone-aware parser output feeds into the LLM prompt (structured representation, not raw XML)
- Placement instructions are validated before application (text substring check)
- Manual retry only on failure (consistent with existing LLM error policy)

## Boundaries

- **No UI changes** — mapping table, wizard flow, PDF viewer all stay exactly as-is
- **No direct DOCX generation by LLM** — LLM produces instructions, sanitization service applies them
- **No incremental diffing** — always full regeneration, no change tracking complexity
- **Remove blue `.mapped-highlight` CSS** — unused visual noise added in previous commit, serves no purpose

## Acceptance Criteria

- User edits mappings in table, clicks regenerate, document is produced with correct placeholders (no corruption)
- Initial upload auto-map uses the same pipeline as regeneration
- KB receives structural context from LLM placement decisions
- Mismatched instructions are skipped with user-visible warnings
- Old mechanical `apply-from-mapping` replacement logic removed
- Mapping table UI completely unchanged

## Decisions Made

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Full regeneration (entire mapping table) | LLM needs full picture for correct placement; no diffing complexity |
| 2 | Structured zone map + paragraph text as LLM input | Zone-aware parser already exists; balances structure vs token cost |
| 3 | Placement instructions (JSON) as LLM output | Safer — validate before touching document; LLM doesn't touch DOCX directly |
| 4 | Skip and warn on mismatches | Apply what works, show warnings for failures; user fixes in table |
| 5 | Replace mechanical approach entirely | One code path; old approach is fundamentally broken for user-added entries |
| 6 | Unify auto-map and regeneration pipelines | Same LLM placement logic for both; consistent behavior, less code |
| 7 | Enrich KB from LLM placement decisions | Zone + marker type context improves future auto-map accuracy |
