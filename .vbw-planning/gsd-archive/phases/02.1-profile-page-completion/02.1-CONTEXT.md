## Phase Boundary

The profile page already exists at `/profile` with a routed component.

</domain>

<decisions>
## Implementation Decisions

### Profile data fields
- Minimal data model: add display name (optional) and avatar to user
- No email, phone, bio, or other personal information fields
- Display name is optional — username always available as fallback
- Display name not required during onboarding (no changes to onboarding wizard)
- Avatar: upload with initials fallback (auto-generated circle from display name or username initials when no image uploaded)
- Avatar and display name appear on profile page and header bar only — sidebar keeps username, audit logs use username

### Security section
- Security settings inline on the same profile page — not tabs or separate page
- One scrollable page: Account Info section at top, Security Settings section below
- Password change via modal dialog (reuse existing PasswordChange component)
- TOTP management: view status, regenerate. Cannot disable — 2FA is mandatory
- Only admin can reset a user's TOTP (existing admin panel functionality)
- No active sessions list on profile page (admin-only concern)

### Profile editing UX
- View/edit toggle: fields display as text by default, "Edit Profile" button switches to input mode
- Save and Cancel buttons in edit mode
- Avatar upload only available in edit mode — shows camera/upload icon overlay on avatar
- Toast notification on save: "Profile updated" — fields return to view mode
- No unsaved changes warning needed (minimal fields)

### Account metadata
- Show join date ("Member since"), last login timestamp, and user role (Admin/User)
- No usage statistics — keep it clean

### Claude's Discretion
- Avatar crop/resize approach (simplest that produces good results)
- Account metadata placement (near avatar area vs separate card — fit existing layout)
- Exact spacing, typography, and card styling
- Avatar file size/type validation limits
- Error state handling for failed uploads or save operations

</decisions>
