## Summary

This phase completes the skeleton Profile page by adding avatar upload with initials fallback, display name field, view/edit mode toggle, and wiring up security settings. The existing codebase uses React 19, React Hook Form with Zod validation, TanStack Query for data management, and Radix UI components (via shadcn/ui patterns). The backend is Express + Prisma with SQLite.

Key technical decisions: Store avatar URLs (not base64 blobs) in database, use native HTML5 file input for simplicity (no heavy cropping library needed for MVP), implement FormData upload with CSRF protection, and leverage existing toast notifications (Sonner) for user feedback. The profile editing pattern uses view/edit toggle state with optimistic updates via TanStack Query mutations.

**Primary recommendation:** Use native browser file input with client-side validation, multer for Express file handling, store avatars as files with URL references in database, and implement view/edit toggle with React Hook Form. Reuse existing PasswordChange and TOTPSetup components in Radix UI Dialog modals.

### Pattern 1: View/Edit Toggle Form
**What:** Form displays as read-only text by default, switches to editable inputs on "Edit" button click
**When to use:** Profile pages, settings pages with minimal fields
**Example:**
```typescript
// Pattern from community best practices
function ProfileForm() {
  const [isEditing, setIsEditing] = useState(false);
  const { user } = useAuth();
  const updateProfile = useUpdateProfile();

  const { register, handleSubmit, reset } = useForm({
    defaultValues: { displayName: user?.displayName || '' }
  });

  const onSave = async (data) => {
    await updateProfile.mutateAsync(data);
    setIsEditing(false);
    toast.success('Profile updated');
  };

  const onCancel = () => {
    reset();
    setIsEditing(false);
  };

  return (
    <form onSubmit={handleSubmit(onSave)}>
      {isEditing ? (
        <>
          <Input {...register('displayName')} />
          <Button type="submit">Save</Button>
          <Button onClick={onCancel}>Cancel</Button>
        </>
      ) : (
        <>
          <p>{user?.displayName || 'Not set'}</p>
          <Button onClick={() => setIsEditing(true)}>Edit Profile</Button>
        </>
      )}
    </form>
  );
}
```

const handleAvatarUpload = async (e: React.ChangeEvent<HTMLInputElement>) => {
  const file = e.target.files?.[0];
  if (!file) return;

### Pattern 3: FormData Upload with CSRF
**What:** Include CSRF token in FormData for multipart/form-data uploads
**When to use:** File uploads to CSRF-protected Express endpoints
**Example:**
```typescript
// Extend existing apiClient for FormData
export async function apiUpload<T>(
  endpoint: string,
  formData: FormData
): Promise<T> {
  await ensureCsrfToken();
  const csrfToken = getCsrfToken();

  const headers: Record<string, string> = {};
  if (csrfToken) {
    headers['X-CSRF-Token'] = csrfToken;
  }

  // Don't set Content-Type — browser sets it with boundary
  const response = await fetch(`${API_BASE_URL}${endpoint}`, {
    method: 'POST',
    headers,
    body: formData,
    credentials: 'include'
  });

  // ... error handling same as apiClient
}
```

### Anti-Patterns to Avoid
- **Storing base64 images in database:** 33% size overhead, poor performance for images >1KB. Use file storage + URL.
- **Not validating file size/type server-side:** Client validation can be bypassed. Always validate on server.
- **Using Content-Type header with FormData:** Browser automatically sets multipart/form-data with boundary. Manual header breaks upload.
- **Complex cropping for MVP:** User can crop locally before upload. Don't add react-cropper unless specifically requested.
- **Forgetting CSRF token on FormData uploads:** CSRF protection applies to file uploads too. Include X-CSRF-Token header.

## Common Pitfalls

### Pitfall 1: CSRF Token Missing on FormData Upload
**What goes wrong:** File upload returns 403 Forbidden with "Invalid CSRF token" error
**Why it happens:** Content-Type header set manually, or CSRF token not included in FormData/headers
**How to avoid:** Include CSRF token in X-CSRF-Token header, don't manually set Content-Type (browser handles it)
**Warning signs:** Works with JSON body but fails with FormData; 403 specifically on file upload endpoint

### Pitfall 2: Avatar Flash During Load
**What goes wrong:** Initials fallback briefly flashes before avatar image loads
**Why it happens:** AvatarImage renders immediately, causing layout shift when fallback is replaced
**How to avoid:** Use Radix Avatar's built-in loading state handling or add delayMs to AvatarFallback
**Warning signs:** Visual jank on profile page load; fallback briefly visible even when avatar exists

### Pitfall 3: File Size Bomb
**What goes wrong:** Users upload huge images (10MB+), slow down server, fill disk
**Why it happens:** No server-side file size validation, trusting client-side checks only
**How to avoid:** Configure multer limits: `{ fileSize: 2 * 1024 * 1024 }` for 2MB max
**Warning signs:** Disk usage grows rapidly; slow upload performance; OOM errors

### Pitfall 4: MIME Type Spoofing
**What goes wrong:** Users upload executable files disguised as images (e.g., virus.jpg.exe)
**Why it happens:** Trusting file extension or client-provided MIME type without validation
**How to avoid:** Use multer fileFilter to check MIME type server-side; consider magic number validation for extra security
**Warning signs:** Unexpected file types in uploads directory; security scanner alerts

### Pitfall 5: Stale Avatar Cache
**What goes wrong:** Avatar update succeeds but old image still displays
**Why it happens:** TanStack Query cache not invalidated after avatar upload; browser cache
**How to avoid:** Invalidate user query on upload success; append timestamp to avatar URL for cache busting
**Warning signs:** Refresh page shows new avatar but not immediately after upload; other users see old avatar

### Pitfall 6: React Hook Form File Input
**What goes wrong:** File input value doesn't register with React Hook Form
**Why it happens:** File inputs return FileList, not File or string. Requires special handling.
**How to avoid:** Don't use register() for file inputs; use manual onChange handler and ref
**Warning signs:** Form submits but file is undefined; validation errors on file field

### Pitfall 7: Prisma SQLite BLOB Limitation
**What goes wrong:** Attempting to store binary data directly in Prisma SQLite
**Why it happens:** Prisma doesn't support SQLite BLOB type (as of 2026)
**How to avoid:** Store files on disk, save URL/path in database as String field
**Warning signs:** Prisma schema errors on Bytes type with SQLite; documentation says BLOB unsupported

export async function uploadAvatar(formData: FormData): Promise<{ avatarUrl: string }> {
  // Use special upload function for FormData
  return apiUpload<{ avatarUrl: string }>('/api/profile/avatar', formData);

// frontend/src/features/profile/hooks.ts
export function useUploadAvatar() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: uploadAvatar,
    onSuccess: () => {
      // Invalidate user query to refetch with new avatar
      queryClient.invalidateQueries({ queryKey: ['auth', 'me'] });
    },
    onError: (error: ApiError) => {
      toast.error(error.message || 'Failed to upload avatar');
    }
  });
}

// frontend/src/routes/Profile.tsx
const uploadAvatar = useUploadAvatar();

const handleAvatarChange = async (e: React.ChangeEvent<HTMLInputElement>) => {

    toast.error('Image must be less than 2MB');
    return;

  await uploadAvatar.mutateAsync(formData);
};

### Multer Backend Configuration
```typescript
// backend/src/routes/profile.ts
import multer from 'multer';
import path from 'path';
import fs from 'fs/promises';

// Configure multer storage
const storage = multer.diskStorage({
  destination: async (req, file, cb) => {
    const uploadDir = path.join(process.cwd(), 'uploads', 'avatars');
    await fs.mkdir(uploadDir, { recursive: true });
    cb(null, uploadDir);
  },
  filename: (req, file, cb) => {
    const session = req.session as any;
    const ext = path.extname(file.originalname);
    cb(null, `${session.userId}${ext}`); // One avatar per user
  }
});

// File filter
const fileFilter = (req: any, file: any, cb: any) => {
  const allowedTypes = ['image/jpeg', 'image/png', 'image/gif'];
  if (allowedTypes.includes(file.mimetype)) {
    cb(null, true);
  } else {
    cb(new Error('Only JPEG, PNG, and GIF images allowed'), false);
  }
};

const upload = multer({
  storage,
  fileFilter,
  limits: { fileSize: 2 * 1024 * 1024 } // 2MB
});

// Route
router.post('/avatar', requireAuth, upload.single('avatar'), async (req, res) => {
  try {
    const session = req.session as any;
    const avatarUrl = `/uploads/avatars/${req.file?.filename}`;

    // Update user record
    await prisma.user.update({
      where: { id: session.userId },
      data: { avatarUrl }
    });

    res.json({ avatarUrl });
  } catch (error) {
    console.error('[profile] Error uploading avatar:', error);
    res.status(500).json({ error: 'Failed to upload avatar' });
  }
});
```

## Open Questions

1. **Should we implement server-side image resize/optimization?**
   - What we know: User can upload up to 2MB images; avatars displayed at 96px typically
   - What's unclear: Performance impact of serving full-size images vs resize overhead
   - Recommendation: Start without resize (simpler), add if performance metrics show issue. Sharp library is standard if needed.

2. **Avatar cache busting strategy?**
   - What we know: Browser/CDN cache can show stale avatars
   - What's unclear: Best balance between cache performance and freshness
   - Recommendation: Append `?t={updatedAt}` timestamp to avatar URL; invalidate on upload

3. **Should lastLoginAt update on every request or just login?**
   - What we know: Need to display "Last login" on profile page
   - What's unclear: Session validation vs actual login tracking
   - Recommendation: Update only on successful login (not session validation) for cleaner semantics
