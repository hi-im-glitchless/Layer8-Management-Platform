  <action>
    1. Create backend/src/services/llm/audit.ts:
       - `logLLMInteraction(userId, ipAddress, data)`: Logs LLM interaction to audit trail
       - Uses existing `logAuditEvent()` from services/audit.ts (reuse Phase 1 hash-chain infrastructure)
       - Data includes: `{ promptSanitized: string, responseFull: string, inputTokens: number, outputTokens: number, model: string }`
       - Action name: `'llm.generate'`
       - Details object stores: model, inputTokens, outputTokens, promptLength, responseLength, promptSanitized (full text), responseFull (full text)
       - NOTE: Do NOT log provider used (per user decision -- not relevant for auditing)
       - NOTE: Store full prompt and full response (GDPR compliance -- proof of what LLM saw)

    2. Create backend/src/routes/llm.ts:
       - POST /generate endpoint (will be mounted at /api/llm):
         - Requires authentication (requireAuth middleware)
         - Accepts JSON body: `{ prompt: string, systemPrompt?: string, maxTokens?: number }`
         - Validate prompt is non-empty string
         - Set SSE headers: Content-Type: text/event-stream, Cache-Control: no-cache, Connection: keep-alive, X-Accel-Buffering: no
         - Call res.flushHeaders() immediately
         - Create AbortController, listen for req.on('close') to abort (client disconnect)
         - Call createLLMClient() then client.generateStream()
         - For each chunk from async generator:
           - If chunk.text: send `event: delta\ndata: {"text":"..."}\n\n`
           - Track full response (concatenate text chunks)
         - On stream complete (chunk.done=true): send `event: done\ndata: {"usage":{"inputTokens":N,"outputTokens":N}}\n\n` then res.end()
         - After stream ends successfully: call logLLMInteraction with full prompt, full response, usage, model
         - On error: send `event: error\ndata: {"message":"...","retryable":true}\n\n` then res.end()
           - Still log the interaction (with partial response if any) for audit trail
           - Error message should be technical and actionable (per user decision)
         - Use try/catch around the entire stream loop
         - Handle req 'close' event (client disconnected): abort the stream, still log partial interaction

    3. Extend backend/src/routes/admin.ts with LLM settings and status routes:
       - GET /llm-settings: Returns current LLM settings (getLlmSettings) -- mask anthropicApiKey (show only last 4 chars)
       - PUT /llm-settings: Updates LLM settings (updateLlmSettings) with body validation via zod
         - Validate: defaultModel, templateAdapterModel, executiveReportModel are optional non-empty strings; anthropicApiKey is optional string; fallbackEnabled is boolean
         - Log setting changes to audit trail: logAuditEvent with action 'admin.llm.settings.update'
       - GET /llm-status: Returns provider status
         - Call createLLMClient() then client.checkStatus()
         - Returns array of { provider, available, error? }
       - POST /llm-start-cliproxy: Attempts to start CLIProxyAPI
         - Use child_process.spawn() to start CLIProxyAPI process
         - This is best-effort -- if CLIProxyAPI is a systemd service, try `systemctl start cliproxy` or similar
         - If CLIProxyAPI binary path is unknown, return instructions to start manually
         - For now, implement as a health check + instruction response (actual start mechanism depends on deployment)
         - Return { success: boolean, message: string }
       - All new routes use requireAdmin middleware (already applied via router.use)

    4. Update backend/src/index.ts:
       - Import llmRouter from './routes/llm.js'
       - Mount: `app.use('/api/llm', llmRouter)` (after session and CSRF middleware, before general rate limiter or with its own rate limit)
       - NOTE: The SSE endpoint must NOT have response buffering middleware. Express json() middleware won't interfere since we set headers manually. Verify no compression middleware is applied globally.
