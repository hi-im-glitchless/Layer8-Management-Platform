# Phase 02.1 Plan 01: Backend Infrastructure for Profile Management Summary

**One-liner:** Extended User model with profile fields (displayName, avatarUrl, lastLoginAt), created profile API endpoints with avatar upload, enabled TOTP regeneration for authenticated users, and added frontend apiUpload helper.

## What Was Built

### Backend Changes

1. **User Model Extensions** (Prisma schema):
   - Added `displayName String?` - optional display name (max 50 chars)
   - Added `avatarUrl String?` - path to avatar image with cache-busting
   - Added `lastLoginAt DateTime?` - timestamp of last successful login

2. **Profile API Routes** (`backend/src/routes/profile.ts`):
   - `PUT /api/profile` - Update displayName (validated with Zod, max 50 chars, trimmed)
   - `POST /api/profile/avatar` - Upload avatar with multer (2MB limit, image-only, diskStorage)
   - `DELETE /api/profile/avatar` - Delete avatar file and clear database reference
   - All routes protected with `requireAuth` middleware
   - Audit logging for all profile operations

3. **Auth Route Enhancements** (`backend/src/routes/auth.ts`):
   - **GET /api/auth/me**: Now returns `displayName`, `avatarUrl`, `lastLoginAt`, `createdAt` in addition to existing fields
   - **POST /api/auth/login**: Updates `lastLoginAt` timestamp on successful password verification
   - **POST /api/auth/totp/setup**: Modified to allow TOTP re-setup for fully authenticated users (when `totpVerified=true`), enabling TOTP regeneration from profile page while preventing re-setup during initial login flow

4. **Infrastructure** (`backend/src/index.ts`):
   - Mounted profile router at `/api/profile`
   - Added static file serving for `/uploads` path (before rate limiting for performance)
   - Created `uploads/avatars/` directory on startup

5. **File Upload Implementation**:
   - Multer diskStorage saves to `uploads/avatars/{userId}.{ext}`
   - FileFilter allows only `image/jpeg`, `image/png`, `image/gif`
   - 2MB file size limit
   - Automatic cleanup of old avatar when extension changes
   - Cache-busting URLs: `/uploads/avatars/{filename}?t={timestamp}`

### Frontend Changes

1. **User Type Extensions** (`frontend/src/features/auth/types.ts`):
   - Added `displayName?: string | null`
   - Added `avatarUrl?: string | null`
   - Added `createdAt?: string`
   - Added `lastLoginAt?: string | null`

2. **Upload Helper** (`frontend/src/lib/api.ts`):
   - Added `apiUpload<T>(endpoint, formData)` function
   - Mirrors `apiClient` logic for FormData uploads
   - Includes CSRF token support via `ensureCsrfToken()`
   - Does NOT set Content-Type (browser handles multipart boundary)
   - Standard error handling and 401 redirect

## Key Implementation Details

### TOTP Regeneration Logic
The TOTP setup endpoint now implements conditional logic:
- If `user.totpEnabled` is true AND `session.totpVerified` is true: Allow re-setup (authenticated user regenerating from profile)
- If `user.totpEnabled` is true AND `session.totpVerified` is NOT true: Return 400 (prevents re-setup during login flow)
- If `user.totpEnabled` is false: Allow setup (first-time TOTP setup)

This ensures only authenticated users who have already verified their current TOTP can regenerate it, per the locked decision that 2FA is mandatory and cannot be disabled.

### Avatar Upload Flow
1. User uploads image via `POST /api/profile/avatar`
2. Multer validates file type and size
3. File saved as `uploads/avatars/{userId}.{ext}` (overwrites same extension)
4. If extension changed, old avatar file deleted
5. Database updated with `/uploads/avatars/{filename}?t={timestamp}`
6. Cache-busting timestamp invalidates browser cache

### Static File Serving Order
Static `/uploads` route mounted BEFORE rate limiting to ensure fast avatar delivery without hitting rate limits.
