  <name>Task 1: Add DenyListTerm model to Prisma schema and create deny list service</name>
  <files>
    backend/prisma/schema.prisma
    backend/src/services/denyList.ts
  </files>
  <action>
    **backend/prisma/schema.prisma** -- Add a new model:
    ```prisma
    model DenyListTerm {
      id          String   @id @default(cuid())
      term        String
      description String?
      createdBy   String?
      isActive    Boolean  @default(true)
      createdAt   DateTime @default(now())
      updatedAt   DateTime @updatedAt

      creator User? @relation(fields: [createdBy], references: [id], onDelete: SetNull)

      @@unique([term])
      @@index([isActive])
    }
    ```
    Also add the reverse relation to the User model: `denyListTerms DenyListTerm[]`

    Run `npx prisma db push` to apply the schema change.

    **backend/src/services/denyList.ts:**
    - `getAllActiveTerms()`: Returns all active deny list terms (term strings only) as `string[]`. This is the hot path -- called on every sanitize request. Query: `where: { isActive: true }`, select only `term` field.
    - `listTerms(options?: { includeInactive?: boolean })`: Returns all terms with full details (id, term, description, createdBy, isActive, createdAt, updatedAt, creator username). For admin UI listing.
    - `createTerm(term: string, description: string | null, createdBy: string)`: Creates a new deny list term. Trim whitespace from term. If term already exists (unique constraint), throw a descriptive error. Return the created record.
    - `updateTerm(id: string, data: { term?: string, description?: string | null, isActive?: boolean })`: Updates a deny list term by ID. Return the updated record.
    - `deleteTerm(id: string)`: Hard-delete a deny list term by ID. Return true/false.
    - `bulkCreateTerms(terms: Array<{ term: string, description?: string }>, createdBy: string)`: Create multiple terms at once, skip duplicates (use skipDuplicates or try/catch per term). Return count of created terms.

    Import prisma from `@/db/prisma.js`. Follow existing service patterns (see auth.ts, session.ts).
  </action>

    2. `npx prisma studio` shows DenyListTerm table exists (or `npx prisma db pull` confirms schema)
    3. TypeScript compiles: `npx tsc --noEmit` shows no errors in denyList.ts
  </verify>
  <done>
    DenyListTerm model exists in Prisma schema with unique term constraint. Service provides CRUD operations plus `getAllActiveTerms()` for the sanitization hot path. Schema is pushed to SQLite.

  <action>
    **backend/src/routes/denyList.ts:**
    Create an Express router with admin-only endpoints:

    - `GET /api/deny-list` -- List all deny list terms. Query params: `includeInactive=true` (optional). Response: `{ terms: DenyListTerm[] }`. Requires authentication + admin role.
    - `POST /api/deny-list` -- Create a new term. Body: `{ term: string, description?: string }`. Validate with zod: term is required, non-empty, max 200 chars. Response: 201 with `{ term: DenyListTerm }`. Requires admin.
    - `PUT /api/deny-list/:id` -- Update a term. Body: `{ term?: string, description?: string, isActive?: boolean }`. Response: `{ term: DenyListTerm }`. Requires admin.
    - `DELETE /api/deny-list/:id` -- Delete a term. Response: 204 No Content. Requires admin.
    - `POST /api/deny-list/bulk` -- Bulk create terms. Body: `{ terms: Array<{ term: string, description?: string }> }`. Validate: terms is array, max 100 items. Response: `{ created: number, skipped: number }`. Requires admin.
    - `GET /api/deny-list/active` -- Get active terms only (term strings). Response: `{ terms: string[] }`. This endpoint is for the sanitization service proxy and requires authentication but NOT admin (any authenticated user needs this during sanitization). This endpoint should be efficient -- it's called on every sanitize request.

    Use the existing auth middleware pattern from other routes (check `req.session.userId` and `req.session.isAdmin`). Follow the pattern in `backend/src/routes/admin.ts` for admin-only guards.

    Add audit logging for create, update, and delete operations using the existing audit service (action: "deny_list.create", "deny_list.update", "deny_list.delete").

    **backend/src/index.ts:**
    - Import `denyListRouter` from `./routes/denyList.js`
    - Mount: `app.use('/api/deny-list', denyListRouter);` after the admin routes
