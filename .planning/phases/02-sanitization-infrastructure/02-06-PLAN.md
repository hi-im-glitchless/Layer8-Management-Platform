<task type="auto">
  <name>Task 1: Fix Presidio model loading with custom SpacyNlpEngine</name>
  <files>sanitization-service/app/services/sanitizer.py</files>
  <action>
The bug: `SanitizationService.__init__` creates a `NlpEngineProvider` with `nlp_model.meta["name"]` as the model name. For models installed via `spacy download`, `.meta["name"]` returns the base name (e.g., `core_web_lg`) without the language prefix. Presidio then calls `spacy.cli.download("core_web_lg")` which fails because no such package exists.

The fix: Create a `PreloadedSpacyNlpEngine` class that subclasses `presidio_analyzer.nlp_engine.SpacyNlpEngine` and injects already-loaded spaCy models directly, bypassing Presidio's download-and-load mechanism entirely. This is cleaner than trying to fix the model name string because:
1. Models are already loaded in main.py's lifespan (no point loading them twice)
2. It avoids any future naming inconsistencies between spaCy versions

Implementation:

1. Add import: `from presidio_analyzer.nlp_engine import SpacyNlpEngine, NerModelConfiguration`

2. Create class `PreloadedSpacyNlpEngine(SpacyNlpEngine)` with:
   - `__init__(self, loaded_models: dict[str, Any])` that:
     - Builds the `models` list as `[{"lang_code": lang, "model_name": lang} for lang in loaded_models]` (model_name is irrelevant since we skip loading)
     - Calls `super().__init__(models=models, ner_model_configuration=NerModelConfiguration())`
     - Sets `self.nlp = loaded_models` directly (this is the dict SpacyNlpEngine.load() would create)
   - Override `load(self)` as a no-op (`pass`) since models are already loaded

3. Rewrite `SanitizationService.__init__` to use `PreloadedSpacyNlpEngine`:
   - Remove the per-language for loop that creates NlpEngineProvider instances
   - Create ONE `PreloadedSpacyNlpEngine(nlp_models)` instance
   - Create ONE `AnalyzerEngine(nlp_engine=nlp_engine, supported_languages=list(nlp_models.keys()))`
   - Register all custom recognizers on this single analyzer
   - Store `self.analyzers` as before but mapping each language to the SAME analyzer instance (or simplify to `self.analyzer = analyzer` and update `sanitize()` method to use `self.analyzer` instead of `self.analyzers.get(language)`)

4. Update `sanitize()` method:
   - Replace `analyzer = self.analyzers.get(language)` / fallback logic with the single analyzer
   - The single analyzer supports all languages, so just check language is in `self.nlp_models` and fall back to "en" if not

Keep `desanitize()` and `_merge_overlapping_results()` unchanged.

Do NOT modify main.py -- the model loading in lifespan is correct as-is.

<task type="auto">
  <name>Task 2: Create Dockerfile.test for running tests in Docker</name>
  <files>sanitization-service/Dockerfile.test, sanitization-service/.dockerignore</files>
  <action>
The bug: The production `.dockerignore` excludes `tests/` and the `Dockerfile` only copies `app/`. Tests cannot run in the production image, and they should not -- the production image should stay lean.

The fix: Create a separate `Dockerfile.test` that extends the production image pattern but includes test files. Do NOT modify the production Dockerfile or remove `tests/` from `.dockerignore`.

1. Create `sanitization-service/Dockerfile.test`:
   ```dockerfile
   # Test image for running pytest inside Docker
   # Uses same base as production but includes test files and dependencies
   FROM python:3.12-slim

   # Install system dependencies (same as production)
   RUN apt-get update && \
       apt-get install -y --no-install-recommends \
       gcc g++ build-essential curl \
       && rm -rf /var/lib/apt/lists/*

   WORKDIR /app

   # Copy requirements and install (same as production)
   COPY requirements.txt .
   RUN pip install --no-cache-dir -r requirements.txt

   # Download spaCy models -- use BOTH large (production) and small (test) models
   # Large models needed for service startup, small models used in conftest.py
   RUN python -m spacy download en_core_web_lg && \
       python -m spacy download pt_core_news_lg && \
       python -m spacy download en_core_web_sm && \
       python -m spacy download pt_core_news_sm

   # Copy application code
   COPY app/ ./app/

   # Copy test files and config
   COPY tests/ ./tests/
   COPY pyproject.toml .

   # Create non-root user
   RUN useradd -m appuser && chown -R appuser:appuser /app
   USER appuser

   # Default command: run pytest
   CMD ["python", "-m", "pytest", "tests/", "-v"]
   ```

2. Modify `sanitization-service/.dockerignore`: Remove the `tests/` line so Docker build context includes tests when building with `Dockerfile.test`. The production `Dockerfile` explicitly uses `COPY app/ ./app/` so it will NOT accidentally include tests even without the dockerignore exclusion. The `.dockerignore` after edit should be:
   ```
   __pycache__
   *.pyc
   .env
   .venv
   .pytest_cache
   *.log
   .git
   .gitignore
   ```

Note: Removing `tests/` from `.dockerignore` is safe because the production `Dockerfile` uses explicit `COPY app/ ./app/` (not `COPY . .`), so tests never enter the production image regardless. But `Dockerfile.test` needs `tests/` in the build context.
  </action>
