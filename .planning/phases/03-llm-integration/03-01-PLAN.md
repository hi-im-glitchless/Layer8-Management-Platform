<task type="auto">
  <name>Task 2: LLM provider implementations, client with fallback, and retry utility</name>
  <files>

  <action>
    1. Create backend/src/services/llm/retry.ts:
       - Use `exponential-backoff` package (backOff function)
       - `retryWithBackoff<T>(fn, config?)`: Wraps async function with retry
       - Default config: maxAttempts=3, startingDelay=500ms, maxDelay=10000ms, jitter='full'
       - Only retry on transient errors: 503, 429, timeout, ECONNREFUSED
       - Log retry attempts with attempt number

    2. Create backend/src/services/llm/providers/cliproxy.ts:
       - `CLIProxyProvider` class with:
         - Constructor takes baseUrl (defaults to http://localhost:8080) -- uses OpenAI SDK with custom baseURL
         - `stream(messages: LLMMessage[], model: string, maxTokens?: number)`: Returns AsyncGenerator<LLMStreamChunk>
         - Uses `new OpenAI({ baseURL, apiKey: 'not-needed' })` -- CLIProxyAPI doesn't require API key
         - Streaming via `openai.chat.completions.create({ stream: true })` with async iterator
         - Yields `{ text, done: false }` for each content delta, then `{ text: '', done: true, usage }` at end
         - Track usage from final chunk (chunk.usage for prompt_tokens/completion_tokens)
         - Accept AbortSignal for stream cancellation: pass signal option to SDK create call
       - `healthCheck()`: Attempts a lightweight request to verify CLIProxyAPI is reachable (fetch baseUrl/models or similar)

    3. Create backend/src/services/llm/providers/anthropic.ts:
       - `AnthropicProvider` class with:
         - Constructor takes apiKey
         - `stream(messages: LLMMessage[], model: string, maxTokens?: number)`: Returns AsyncGenerator<LLMStreamChunk>
         - Uses `new Anthropic({ apiKey })`
         - Streaming via `client.messages.create({ stream: true })` with async iterator
         - Handle event types: 'content_block_delta' with 'text_delta' for text, 'message_delta' for usage
         - Convert LLMMessage[] format: extract system message separately (Anthropic requires system as top-level param, not in messages array)
         - Yields same LLMStreamChunk format as CLIProxy provider
         - Accept AbortSignal for cancellation
       - `healthCheck()`: Validates API key is set and non-empty (no actual API call needed)

    4. Create backend/src/services/llm/client.ts:
       - `LLMClient` class:
         - Constructor takes settings from getLlmSettings()
         - Instantiates CLIProxyProvider (always, using cliproxyBaseUrl from settings)
         - Instantiates AnthropicProvider only if anthropicApiKey is set and fallbackEnabled
         - `generateStream(messages: LLMMessage[], options?: { maxTokens?, signal? })`: AsyncGenerator<LLMStreamChunk>
           - Try CLIProxyProvider first
           - On failure: if AnthropicProvider configured, try it
           - If both fail (or only CLIProxy configured and it fails): throw descriptive error
           - Error message: "CLIProxyAPI connection refused. Check that the service is running, or switch to API fallback in Settings." (per user decision -- technical, actionable)
         - `checkStatus()`: Returns array of LLMProviderStatus (one per configured provider)
       - `createLLMClient()` factory function: reads settings, returns configured LLMClient instance
       - NOTE: Do NOT log which provider was used (per user decision -- provider not relevant for auditing)
  </action>
